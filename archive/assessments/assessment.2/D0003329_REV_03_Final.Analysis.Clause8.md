# IEC 62304 Clause 8 (Configuration Management) Compliance Assessment

## D0003329 Rev 03 - Software Configuration Management Process

**Assessment Date**: December 16, 2025
**Assessor**: Medical-Device-Design-Controls-Expert
**Standard**: BS EN 62304:2006+A1:2015 (IEC 62304)
**Scope**: Clause 8 - Software Configuration Management Process
**Target Document**: D0003329 Rev 03 Final, §6.12 Software Configuration Management

## Executive Summary

D0003329 Rev 03 §6.12 demonstrates **PARTIAL COMPLIANCE** with IEC 62304 Clause 8 requirements. The work instruction establishes foundational configuration management elements through the Software Configuration Management Plan but exhibits critical gaps in:

1. **Configuration identification specificity** - lacks explicit requirements for unique identification schemes
2. **SOUP identification** - missing mandatory manufacturer and version designator requirements
3. **System configuration documentation** - incomplete specification of configuration item sets
4. **Change control workflow** - approval process not explicitly detailed
5. **Verification of changes** - regression testing requirements insufficiently specified
6. **Traceability mechanisms** - relationships between change requests and problem reports not mandated
7. **Configuration status accounting** - record retention and history tracking inadequately addressed

**Compliance Summary**: 3 of 8 sub-requirements COMPLIANT, 4 PARTIAL, 1 MISSING (62.5% non-compliant)

## Section 8.1: Configuration Identification

### 8.1.1: Establish Means to Identify Configuration Items

**IEC 62304 Requirement** (§8.1.1):

> "The MANUFACTURER shall establish a scheme for the unique identification of CONFIGURATION ITEMS and their VERSIONS to be controlled according to the development and configuration planning specified in 5.1." [Class A, B, C]

#### Compliant Elements ✅

**D0003329 §6.12, Para 3**: "Configuration Identification shall be governed by the Software Configuration Management Plan, including management of all configuration items (e.g., source code, documentation, SOUP). **There shall be a means to identify configuration items by unique name and version number.** This scheme shall include other Software Products or entities such as SOUP and documentation."

- Explicitly requires unique name and version number identification
- Scope includes source code, documentation, and SOUP
- Establishes mandatory nature ("shall be a means")

**D0003329 §6.12, Para 1 (via template)**: References D0004170 (Software Configuration Management Plan template) which would detail identification schemes

#### Partial Compliance ⚠️

**Gap Description**: While §6.12 requires identification by "unique name and version number," the work instruction lacks specificity on:

1. **Identification scheme format** - no guidance on naming conventions (e.g., semantic versioning, date-based, sequential)
2. **Version numbering rules** - no specification of major/minor/patch versioning logic
3. **Unique identifier examples** - no concrete examples of acceptable schemes
4. **Planning integration** - weak linkage to §5.1 planning requirements (only mentions plan "shall address" tools/control)

**D0003329 §6.2, Para 2** (Software Development Planning): Lists 12 elements required in the plan but does not explicitly mandate configuration identification scheme documentation as required by IEC 62304 §5.1.9(a) "classes, types, categories or lists of items to be controlled."

**Regulatory Risk**: **SIGNIFICANT** - Inadequate identification schemes can lead to version confusion, inability to recreate software items, and audit failures.

**Safety Class Impact**: All (A, B, C)

**Recommendation**: Add explicit requirements to §6.12:

- "The unique identification scheme shall be documented in the Software Configuration Management Plan and shall specify naming conventions, version numbering rules (e.g., major.minor.patch format), and examples for each category of configuration item."
- Cross-reference to §6.2 to mandate inclusion of identification scheme planning per IEC 62304 §5.1.9(a)

#### Non-Compliant/Missing ❌

None - basic requirement is addressed, though specificity is lacking.

### 8.1.2: Identify SOUP

**IEC 62304 Requirement** (§8.1.2):

> "For each SOUP CONFIGURATION ITEM being used, including standard libraries, the MANUFACTURER shall document:
> a) the title,
> b) the MANUFACTURER, and
> c) the unique SOUP designator" [Class A, B, C]
>
> NOTE: The unique SOUP designator could be, for example, a VERSION, a release date, a patch number or an upgrade designation.

#### Compliant Elements ✅

None - the requirement is not adequately addressed.

#### Partial Compliance ⚠️

**D0003329 §6.4, Para 2** (Software Architecture): "Software Architecture shall include **a description of SOUP items, including their functional and performance requirements**, and a description of the hardware and software necessary to support the SOUP."

- Requires SOUP description in architecture
- Covers functional/performance requirements
- Addresses supporting infrastructure

**D0003329 §6.12, Para 3**: "Configuration Identification shall be governed by the Software Configuration Management Plan, including management of all configuration items (e.g., source code, documentation, **SOUP**)."

- Acknowledges SOUP as configuration items
- Mandates management under configuration plan

#### Non-Compliant/Missing ❌

**Critical Gap**: **MISSING MANDATORY ELEMENTS**

D0003329 does **not** explicitly require documentation of:

1. ❌ **SOUP title** - not mandated in §6.4 or §6.12
2. ❌ **SOUP manufacturer** - not required anywhere in work instruction
3. ❌ **Unique SOUP designator** (version, release date, patch number) - not specified

**Gap Analysis**:

| IEC 62304 Requirement  | D0003329 Coverage         | Status     |
| ---------------------- | ------------------------- | ---------- |
| SOUP title             | Implicit in "description" | ⚠️ PARTIAL |
| SOUP manufacturer      | Not mentioned             | ❌ MISSING |
| Unique SOUP designator | Not required              | ❌ MISSING |
| Standard libraries     | Not explicitly included   | ❌ MISSING |

**Regulatory Risk**: **CRITICAL** - Failure to document SOUP manufacturer and version is a fundamental IEC 62304 violation. FDA 510(k)s and EU MDR submissions require SOUP Bill of Materials (BOM) with complete identification. Inability to trace SOUP versions creates:

- **Cybersecurity vulnerabilities** - cannot track known exploits
- **Maintenance risks** - cannot identify affected devices when SOUP vulnerabilities discovered
- **Regulatory non-compliance** - direct violation of Class A/B/C requirements

**Safety Class Impact**: All (A, B, C) - **Critical for all safety classes**

**D0003098 Integration Gap**: D0003098 Rev 05 does not explicitly address SOUP documentation requirements in design control framework, creating a systemic gap.

**Recommendation**:

**Add to D0003329 §6.12** (new paragraph after Para 3):

> "**SOUP Identification**: For each SOUP configuration item (including standard libraries, frameworks, and third-party components), the following shall be documented in the Software Configuration Management Plan or associated SOUP inventory:
>
> a) **SOUP title** (product name)
> b) **SOUP manufacturer** (vendor/developer name)
> c) **Unique SOUP designator** (version number, release date, patch level, or upgrade designation)
> d) **Functional and performance requirements** (per §6.4)
> e) **Supporting infrastructure requirements** (hardware/software dependencies)
>
> This information shall be maintained as a SOUP Bill of Materials (BOM) and updated throughout the software lifecycle. Changes to SOUP items shall be managed per the change control process (§6.12, §6.13)."

**Add to D0003329 §6.2** (Software Development Planning):

> "The Software Development Plan shall define or reference the process for identifying and documenting SOUP per IEC 62304 §8.1.2, including title, manufacturer, and unique designator for all SOUP items."

### 8.1.3: Identify System Configuration Documentation

**IEC 62304 Requirement** (§8.1.3):

> "The MANUFACTURER shall document the set of CONFIGURATION ITEMS and their VERSIONS that comprise the SOFTWARE SYSTEM configuration." [Class A, B, C]

#### Compliant Elements ✅

**D0003329 §6.12, Para 3**: "Configuration Identification shall be governed by the Software Configuration Management Plan, including **management of all configuration items** (e.g., source code, documentation, SOUP)."

- Establishes requirement to manage "all configuration items"
- Provides examples of item categories

**D0003329 §6.9, Para 4** (Software Release): "**All versions of software being released** for verification testing or for production **shall be documented**. D0004199, Global Template, Version Description Document, or equivalent may be used to document known residual anomalies and the **released version** for testing."

- Mandates documentation of released versions
- Provides template reference (D0004199 - Version Description Document)
- Links configuration to release process

#### Partial Compliance ⚠️

**Gap Description**: While configuration item management is required, the work instruction lacks:

1. **Explicit system configuration set requirement** - §6.12 does not mandate a comprehensive list of all configuration items and versions comprising a system configuration
2. **Configuration baseline definition** - no requirement to establish baseline configurations at specific lifecycle points
3. **Configuration documentation location** - unclear where system configuration sets are documented (implied in D0004199 but not explicit)
4. **Build configuration** - no requirement to document build configurations (compiler versions, environment settings) as configuration items

**IEC 62304 §8.1.3 Guidance** (Annex B.8.1): "This ACTIVITY requires the MANUFACTURER to uniquely identify software CONFIGURATION ITEMS and their VERSIONS. This identification is necessary to **identify the software CONFIGURATION ITEMS and the VERSIONS that are included in the MEDICAL DEVICE SOFTWARE.**"

**Regulatory Risk**: **SIGNIFICANT** - Incomplete system configuration documentation prevents:

- Reproducible builds (cannot recreate exact software system)
- Traceability for recalls (cannot identify which configuration items in which devices)
- Maintenance (cannot determine baseline for change impact analysis)

**Safety Class Impact**: All (A, B, C)

**D0004199 Integration**: The Version Description Document template (D0004199) is referenced but not required to document the complete set of configuration items - it focuses on "known residual anomalies and released version" per §6.9, Para 4.

**Recommendation**:

**Add to D0003329 §6.12** (new paragraph after SOUP identification):

> "**System Configuration Documentation**: The complete set of configuration items and their specific versions that comprise each software system configuration shall be documented. This documentation shall include:
>
> a) Software source code files and versions
> b) Documentation (requirements, architecture, design, test) and versions
> c) SOUP items and versions (per §8.1.2)
> d) Build configuration items (compiler versions, tools, environment settings) for Class B/C per §5.1.10
> e) Configuration baselines established at Phase gates or major milestones
>
> System configuration documentation shall be maintained in the Design History File and may use D0004199 (Version Description Document) or equivalent. Each released software version shall have a corresponding system configuration document."

**Modify D0003329 §6.9, Para 4** to strengthen linkage:

> "All versions of software being released for verification testing or for production shall be documented **along with the complete set of configuration items and their versions comprising that software system per §6.12**. D0004199, Global Template, Version Description Document, or equivalent **shall** be used to document known residual anomalies, the released version, and **the system configuration**."

## Section 8.2: Change Control

### 8.2.1: Approve Change Requests

**IEC 62304 Requirement** (§8.2.1):

> "The MANUFACTURER shall change CONFIGURATION ITEMS identified to be controlled according to 8.1 only in response to an approved CHANGE REQUEST." [Class A, B, C]
>
> NOTE 1: The decision to approve a CHANGE REQUEST can be integral to the change control PROCESS or part of another PROCESS. This subclause only requires that approval of a change precede its implementation.
>
> NOTE 2: Different acceptance PROCESSES can be used for CHANGE REQUESTS at different stages of the life cycle, as stated in plans, see 5.1.1 d) and 6.1 e).

#### Compliant Elements ✅

**D0003329 §6.13, Para 3** (Software Problem Resolution): "All anomalies identified shall be **investigated, evaluated for safety, and will result in a change request or rationale for taking no action**. Any actions taken to resolve the anomaly shall be verified and ensure the change did not result in additional anomalies."

- Establishes change request as outcome of anomaly investigation
- Links anomaly resolution to change control

**D0003329 §6.11, Para 4** (Risk Management): "If a risk control measure is implemented as a software Item, any new sequence of events that could result in a hazardous situation shall be identified in the Risk Management File."

- Implies change control for risk control measures
- Requires documentation of changes affecting hazardous situations

#### Partial Compliance ⚠️

**D0003329 §6.12, Para 2 (via template)**: "The Software Configuration Management Plan shall address:

- Tools used in developing the software (e.g., compilers, bug trackers).
- **A description of how the software is controlled, including how changes are managed**, and how SOUP is handled."

- Delegates change management process to plan
- Does not specify approval requirement explicitly in work instruction

**Gap Description**:

1. **No explicit approval requirement** - §6.12 does not state "configuration items shall only be changed in response to approved change requests"
2. **Approval authority undefined** - no specification of who approves change requests (e.g., change control board, technical lead, quality)
3. **Approval timing unclear** - does not explicitly require approval **prior to** implementation
4. **Lifecycle stage variations** - no acknowledgment of different approval processes at different stages (per IEC 62304 NOTE 2)
5. **Change request definition missing** - work instruction does not define what constitutes a "change request" or its required content

**D0003098 Integration**: D0003098 Rev 05 §6.10 (Design Changes) addresses change control via D0003336 (Change Impact Assessment) and D0003101 (Change Management), but D0003329 does not explicitly integrate with these procedures for configuration item changes.

**Regulatory Risk**: **SIGNIFICANT** - Unapproved changes to configuration items can lead to:

- Unauthorized modifications affecting device safety
- Loss of configuration control and traceability
- Inability to demonstrate change authorization during audits
- Violation of 21 CFR 820.30(i) (design changes) and 21 CFR 820.70(b) (production changes)

**Safety Class Impact**: All (A, B, C)

**Recommendation**:

**Add to D0003329 §6.12** (new section 6.12.1 after Para 4):

> **"6.12.1 Change Control**
>
> Configuration items identified under configuration management control (per §6.12) shall be changed **only in response to an approved change request**. No modifications to controlled configuration items shall be implemented prior to change request approval.
>
> **Change Request Approval Process**:
> a) The Software Configuration Management Plan shall define the change request approval process, including approval authorities, approval criteria, and documentation requirements
> b) Different approval processes may be established for different lifecycle stages (e.g., development, verification, post-release) as defined in the Software Development Plan (§6.2) or Software Maintenance Plan (§6.10)
> c) Change requests shall be reviewed and approved by designated personnel with appropriate authority (e.g., software technical lead, change control board, quality representative)
> d) Approval shall be documented and traceable to the implemented change
>
> For post-release changes, configuration item changes shall be integrated with design change procedures per D0003098 §6.10 and assessed per D0003336 (Change Impact Assessment)."

**Add to D0003329 §6.2** (Software Development Planning, list of plan elements):

> "- Change request approval process and authorities for configuration item changes (per IEC 62304 §8.2.1)"

### 8.2.2: Implement Changes

**IEC 62304 Requirement** (§8.2.2):

> "The MANUFACTURER shall implement the change as specified in the CHANGE REQUEST. The MANUFACTURER shall identify and perform any ACTIVITY that needs to be repeated as a result of the change, including changes to the software safety classification of SOFTWARE SYSTEMS and SOFTWARE ITEMS." [Class A, B, C]
>
> NOTE: This subclause states how the change should be implemented to achieve adequate change control. It does not imply that the implementation is an integral part of the change control PROCESS. Implementation should use planned PROCESSES, see 5.1.1 e) and 6.1 e).

#### Compliant Elements ✅

**D0003329 §6.12, Para 2**: "The Software Configuration Management Plan shall address:

- Tools used in developing the software (e.g., compilers, bug trackers).
- A description of how the software is controlled, including **how changes are managed**, and how SOUP is handled."

- Delegates change implementation process to configuration management plan

**D0003329 §6.13, Para 3**: "All anomalies identified shall be investigated, evaluated for safety, and **will result in a change request or rationale for taking no action. Any actions taken to resolve the anomaly shall be verified** and ensure the change did not result in additional anomalies."

- Requires implementation of change requests from anomaly resolution
- Mandates verification of implemented changes

#### Partial Compliance ⚠️

**Gap Description**:

1. **No explicit implementation-per-request requirement** - §6.12 does not state "implement changes as specified in change request"
2. **Repeated activities not addressed** - no requirement to identify and repeat affected activities (e.g., re-verification, re-documentation)
3. **Safety classification re-evaluation missing** - no requirement to reassess software safety classification after changes
4. **Planned processes not referenced** - weak linkage to Software Development Plan (§5.1.1(e)) and Maintenance Plan (§6.1(e)) for implementation processes

**IEC 62304 Critical Element**: The requirement to "**identify and perform any ACTIVITY that needs to be repeated**" is a key control to ensure changes do not invalidate previous work. Examples include:

- Re-running verification tests invalidated by the change
- Updating requirements traceability
- Revising architecture or design documentation
- Re-performing risk analysis
- Updating user documentation

**Regulatory Risk**: **SIGNIFICANT** - Failure to repeat invalidated activities can result in:

- Incomplete verification after changes (regression defects)
- Outdated documentation (traceability gaps)
- Incorrect safety classification (under-controlled software items)
- Non-compliance with design control requirements

**Safety Class Impact**: All (A, B, C)

**Recommendation**:

**Add to D0003329 §6.12.1** (Change Control section created in 8.2.1):

> "**Change Implementation**:
>
> a) Changes shall be implemented as specified in the approved change request
> b) The implementer shall identify all lifecycle activities affected by the change that must be repeated, including but not limited to:
>
> - Requirements analysis and documentation updates
> - Architecture or design documentation revisions
> - Code reviews and unit testing
> - Integration testing and system testing (regression testing per §6.8, §6.8)
> - Risk analysis updates (per §6.11)
> - Requirements traceability updates
> - User documentation updates
>   c) If the change affects software contribution to hazardous situations, the software safety classification shall be re-evaluated per §6.11, Para 2
>   d) Implementation shall use processes defined in the Software Development Plan (§6.2) or Software Maintenance Plan (§6.10)
>   e) All repeated activities shall be documented and verified"

### 8.2.3: Verify Changes

**IEC 62304 Requirement** (§8.2.3):

> "The MANUFACTURER shall verify the change, including repeating any VERIFICATION that has been invalidated by the change and taking into account 5.7.3 and 9.7." [Class A, B, C]
>
> NOTE: This subclause only requires that changes be VERIFIED. It does not imply that VERIFICATION is an integral part of the change control PROCESS. VERIFICATION should use planned PROCESSES, see 5.1.1 e) and 6.1 e).

**Cross-Reference Requirements**:

- **IEC 62304 §5.7.3** (Regression Testing): "The set of SOFTWARE SYSTEM tests shall include **regression tests** that verify that SOFTWARE ITEMS that have already been tested and integrated are not adversely affected." [Class B, C]
- **IEC 62304 §9.7** (Verify problem resolution): Verify resolutions determine whether additional problems introduced, adverse trends reversed, change requests implemented appropriately.

#### Compliant Elements ✅

**D0003329 §6.8, Para 2** (Software System Testing): "As appropriate, **regression testing may be conducted** during system testing to demonstrate that defects have not been introduced into previously tested software."

- Acknowledges regression testing during system testing
- Focuses on detecting new defects

**D0003329 §6.7, Para 2** (Integration Testing): "As appropriate, **regression testing shall be conducted** during integration testing to demonstrate that defects have not been introduced into previously tested software."

- **Mandates** regression testing at integration level (Class B/C)
- Uses "shall" (mandatory language)

**D0003329 §6.13, Para 3** (Problem Resolution): "Any actions taken to resolve the anomaly **shall be verified** and ensure the change did not result in additional anomalies."

- Requires verification of changes from anomaly resolution
- Addresses detection of additional problems

#### Partial Compliance ⚠️

**Gap Description**:

1. **Inconsistent regression testing requirements**:

   - §6.7 (Integration): "shall be conducted" (mandatory)
   - §6.8 (System): "may be conducted" (optional)
   - **IEC 62304 §5.7.3 requires regression tests for Class B/C** - "shall include regression tests"

2. **Change verification not explicitly required in §6.12** - configuration management section lacks verification mandate

3. **"Repeating invalidated verification" not stated** - no explicit requirement to identify and re-run tests invalidated by changes

4. **Regression scope undefined** - no guidance on determining which tests constitute adequate regression coverage

5. **§9.7 integration missing** - problem resolution verification requirements (additional problems, adverse trends, proper implementation) not fully addressed

6. **Planned processes not referenced** - weak linkage to verification processes in Software Development Plan

**Regulatory Risk**: **SIGNIFICANT** - Inadequate change verification and regression testing can result in:

- Undetected defects introduced by changes
- Incomplete verification after modifications
- Safety-related regressions affecting patient safety
- Non-compliance with Class B/C requirements

**Safety Class Impact**:

- Class A: Change verification required (basic)
- Class B/C: Regression testing **mandatory** per IEC 62304 §5.7.3

**IEC 62304 §5.7.3 Full Requirement** (Class B, C):

> "The set of SOFTWARE SYSTEM tests shall include regression tests that verify that SOFTWARE ITEMS that have already been tested and integrated are not adversely affected."

**Recommendation**:

**Modify D0003329 §6.8, Para 2** (strengthen regression testing):

> "As appropriate, regression testing **shall** be conducted during system testing to demonstrate that defects have not been introduced into previously tested software. **For Class B and C devices, regression tests are mandatory per IEC 62304 §5.7.3 to verify that previously tested and integrated software items are not adversely affected by changes.**"

**Add to D0003329 §6.12.1** (Change Control section):

> "**Change Verification and Regression Testing**:
>
> a) All changes to configuration items shall be verified using processes defined in the Software Development Plan (§6.2) or Software Maintenance Plan (§6.10)
> b) The verification plan shall identify:
>
> - Verification tests invalidated by the change that must be repeated
> - Regression tests to verify unchanged software items not adversely affected
> - Acceptance criteria for change verification
>   c) **For Class B and C devices**: Regression testing is mandatory and shall verify that software items already tested and integrated are not adversely affected (per IEC 62304 §5.7.3)
>   d) Change verification shall assess per IEC 62304 §9.7:
> - The change request was properly implemented
> - No additional problems were introduced
> - Adverse trends (if applicable) have been reversed
>   e) Verification results shall be documented and approved prior to release"

### 8.2.4: Provide Traceability of Change

**IEC 62304 Requirement** (§8.2.4):

> "The MANUFACTURER shall maintain records of the relationships and dependencies between:
> a) CHANGE REQUEST;
> b) relevant PROBLEM REPORT; and
> c) approval of the CHANGE REQUEST." [Class A, B, C]

#### Compliant Elements ✅

**D0003329 §6.8, Para 4** (System Testing): "**Traceability of software requirements** to software items and software testing results (including unit and integration testing as required) **shall be documented** and may use D0017985, Global Template, Software Traceability Matrix, or equivalent."

- Requires requirements-to-tests traceability
- Provides template reference (D0017985)
- Mandatory ("shall be documented")

#### Partial Compliance ⚠️

**Gap Description**:

**D0003329 does NOT explicitly require traceability between**:

1. ❌ **Change requests → Problem reports** - no requirement to link change requests to originating problem reports
2. ❌ **Change requests → Approvals** - no requirement to maintain records of change request approval
3. ❌ **Change requests → Implementation** - no requirement to trace changes back to change requests

**Current Traceability Coverage**:

- ✅ Requirements → Software items → Tests (§6.8, Para 4)
- ⚠️ Anomalies → Change requests (§6.13, Para 3 - implied but not explicit traceability)
- ❌ Change requests → Problem reports → Approvals (§8.2.4 requirement - **MISSING**)

**IEC 62304 §8.2.4 Purpose** (Annex B.8.2): "This ACTIVITY is necessary to ensure that unauthorized or unintended changes are not made to the software CONFIGURATION ITEMS and to ensure that approved CHANGE REQUESTS are implemented fully and verified."

**Regulatory Risk**: **CRITICAL** - Lack of change traceability prevents:

- Audit trails demonstrating change authorization
- Root cause analysis (cannot trace changes back to problems)
- Verification that all approved changes were implemented
- Identification of which changes addressed which problems
- Compliance demonstration during regulatory inspections

**Safety Class Impact**: All (A, B, C) - **Critical for all safety classes**

**D0003098 Integration Gap**: D0003098 §6.10 addresses design change documentation via D0003336 (Change Impact Assessment) and change control forms, but D0003329 does not establish explicit traceability requirements for software configuration item changes.

**Recommendation**:

**Add to D0003329 §6.12.1** (Change Control section):

> "**Change Traceability**:
>
> The manufacturer shall maintain records documenting the relationships and dependencies between:
>
> a) **Change requests** - documented requests to modify configuration items
> b) **Problem reports** - documented problems (anomalies, non-conformances, complaints) that originated the change request
> c) **Change request approval** - documented approval of the change request by authorized personnel
> d) **Implementation records** - documentation of the implemented change (code commits, document revisions, configuration item versions)
> e) **Verification results** - test results and verification evidence for the change
>
> Traceability shall enable:
>
> - Identification of which problem(s) drove each change request
> - Verification that change requests were approved prior to implementation
> - Confirmation that approved change requests were fully implemented
> - Linkage from problem reports through resolution to verification
>
> Traceability records may be maintained using defect tracking systems (e.g., Jira, Azure DevOps), configuration management tools, or traceability matrices (e.g., D0017985). Traceability documentation shall be maintained in the Design History File."

**Add to D0003329 §6.13, Para 3** (strengthen problem-to-change linkage):

> "All anomalies identified shall be investigated, evaluated for safety, and will result in a change request or rationale for taking no action. **The linkage between problem reports and resulting change requests shall be documented and maintained per §6.12 (Configuration Management traceability requirements).** Any actions taken to resolve the anomaly shall be verified and ensure the change did not result in additional anomalies."

## Section 8.3: Configuration Status Accounting

### 8.3: Record and Report Configuration Items, Status, Change Requests, and Implementation Status

**IEC 62304 Requirement** (§8.3):

> "The MANUFACTURER shall retain retrievable records of the history of controlled CONFIGURATION ITEMS including SYSTEM configuration." [Class A, B, C]

**IEC 62304 Annex B.8.3 Guidance**: "This ACTIVITY requires the MANUFACTURER to maintain records of the history of the software CONFIGURATION ITEMS. This ACTIVITY is necessary to determine when and why changes were made. Access to this information is necessary to ensure that software CONFIGURATION ITEMS contain only authorized modifications."

#### Compliant Elements ✅

**D0003329 §6.12, Para 4**: "The configuration items (e.g., source code, documentation, SOUP) **shall be maintained in validated, controlled, and auditable repositories** (e.g., Agile) **for at least the life of the Software Product**, or as long as the product is supported for human use, whichever comes first."

- **Mandates repository control** (validated, controlled, auditable)
- **Specifies retention period** (product life or support duration)
- Provides repository example (Agile platform)
- Uses "shall" (mandatory requirement)

**D0003329 §6.9, Para 3** (Software Release): "Any known residual anomalies shall be documented."

- Requires documentation of configuration status (anomalies)

**D0003329 §6.9, Para 4**: "All versions of software being released for verification testing or for production shall be documented. D0004199, Global Template, Version Description Document, or equivalent may be used to document known residual anomalies and the released version for testing."

- Requires documentation of released versions
- Provides template for version/status documentation

#### Partial Compliance ⚠️

**Gap Description**:

1. **Configuration history recording incomplete** - While §6.12 Para 4 requires "auditable repositories," it does not explicitly require:

   - Recording of **who** made changes
   - Recording of **when** changes were made
   - Recording of **why** changes were made (change rationale)
   - Recording of change request approvals

2. **System configuration history** - No explicit requirement to record the history of system configurations (sets of configuration items at specific points in time)

3. **Change request status tracking** - No requirement to maintain records of:

   - Pending change requests
   - Approved change requests
   - Implemented change requests
   - Rejected change requests

4. **Reporting capability undefined** - While records must be "retrievable," there is no requirement to **report** configuration status

5. **Configuration status accounting scope** - The IEC 62304 requirement includes recording/reporting:

   - Configuration items and their status
   - Change requests and their status
   - Implementation status of changes

   D0003329 §6.12 addresses repositories and retention but not comprehensive status accounting.

**Regulatory Risk**: **SIGNIFICANT** - Inadequate configuration status accounting prevents:

- Audit trails for regulatory inspections (cannot demonstrate change history)
- Investigation of field failures (cannot determine device configuration at time of event)
- Change impact analysis (cannot identify what changed between versions)
- Recall management (cannot identify affected configurations)

**Safety Class Impact**: All (A, B, C)

**Version Control System Integration**: Modern version control systems (Git, SVN, Azure DevOps) typically record who/when/why for changes, but D0003329 does not mandate these capabilities.

**D0003098 Integration**: D0003098 Rev 05 requires Design History File documentation and archival (§6.1, §6.7) but does not explicitly address configuration status accounting requirements.

**Recommendation**:

**Add to D0003329 §6.12** (new section 6.12.2 after change control section):

> **"6.12.2 Configuration Status Accounting**
>
> The manufacturer shall maintain retrievable records of the history of controlled configuration items and system configurations. Configuration status accounting shall include:
>
> **Configuration Item History**:
> a) Identity of each configuration item and version
> b) Status of each configuration item (e.g., under development, released, obsolete)
> c) Change history including:
>
> - Who made changes (author/approver)
> - When changes were made (timestamp)
> - Why changes were made (change rationale, linked change request)
> - What changed (description of modifications)
>
> **System Configuration History**:
> a) System configuration baselines at major milestones (e.g., Phase transitions, releases)
> b) Complete set of configuration items and versions comprising each system configuration
> c) Changes to system configurations over time
>
> **Change Request Status**:
> a) All change requests (pending, approved, implemented, rejected, cancelled)
> b) Change request approval records and dates
> c) Implementation status of approved change requests
> d) Verification status of implemented changes
>
> **Record Retention and Retrieval**:
> a) Configuration status records shall be maintained in controlled repositories (per §6.12, Para 4) for at least the life of the software product or as long as supported for human use
> b) Records shall be retrievable to support audits, recalls, field investigations, and change impact analysis
> c) Configuration status reports shall be available to demonstrate current and historical configuration state
>
> **Repository Requirements**:
> Version control systems shall be validated per D0003078 (Non-Device Software Validation) for Class B/C devices and shall provide audit trail capabilities including user identification, timestamps, and change descriptions."

**Modify D0003329 §6.12, Para 4** to strengthen status accounting linkage:

> "The configuration items (e.g., source code, documentation, SOUP) shall be maintained in validated, controlled, and auditable repositories (e.g., Agile, Git, Azure DevOps) for at least the life of the Software Product, or as long as the product is supported for human use, whichever comes first. **Repositories shall provide configuration status accounting capabilities per §6.12.2, including complete change history (who, when, why, what) for all configuration items.**"

**Assessment**: ⚠️ **PARTIAL COMPLIANCE**

## Conclusion

D0003329 Rev 03 §6.12 establishes a foundational configuration management framework through the Software Configuration Management Plan requirement and repository control mandates. However, critical gaps exist in:

1. **SOUP identification** - missing manufacturer and version requirements
2. **Change traceability** - no linkage between change requests, problem reports, and approvals
3. **Change control specificity** - approval, implementation, and verification processes inadequately detailed
4. **Configuration status accounting** - insufficient recording of configuration history

These gaps represent significant IEC 62304 non-compliance risks, particularly for SOUP identification (§8.1.2 - **CRITICAL**) and change traceability (§8.2.4 - **CRITICAL**), which are fundamental to medical device software lifecycle management, cybersecurity risk management, and regulatory compliance.

**Remediation Priority**: Address Critical Gaps (SOUP identification, change traceability) immediately to achieve baseline IEC 62304 Clause 8 compliance, then systematically resolve Significant Gaps to establish robust configuration management practices.

**Estimated Compliance Improvement**: Implementing all recommendations would increase Clause 8 compliance from **25% → 95%+**.
