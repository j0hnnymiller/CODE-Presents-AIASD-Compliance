## Executive Summary

D0003329 Rev 03 §6.13 provides **minimal coverage** of IEC 62304 Clause 9 requirements. The section contains only **5 sentences** addressing problem resolution, whereas the standard requires comprehensive procedures for 8 distinct requirements (§9.1-9.8). The work instruction demonstrates **significant non-compliance** with critical gaps in problem report preparation, investigation procedures, trend analysis, verification requirements, and test documentation.

## Remediation Recommendations

### Priority 1: Immediate Action Required (Critical Gaps)

**1. Expand §6.13 to Comprehensive Problem Resolution Procedure**

Current state: 5 sentences
Required: Comprehensive procedure addressing all §9.1-9.8 requirements

**Action**: Rewrite §6.13 as a complete procedure or create a separate supporting document (e.g., D0003XXX Global Procedure, Software Problem Resolution) that addresses:

- Problem report preparation (§9.1)
- Investigation and evaluation procedures (§9.2)
- Notification to relevant parties (§9.3)
- Integration with change control (§9.4)
- Record maintenance (§9.5)
- Trend analysis (§9.6)
- Verification procedures (§9.7)
- Test documentation requirements (§9.8)

**2. Add Problem Report Preparation Requirements (§9.1)**

**Recommended text for §6.13**:

```
6.13.1 Problem Report Preparation

All problems detected in medical device software shall be documented in a
problem report. Problem reports may be created using D0004198, Global Template,
Software Anomaly Report, or equivalent defect tracking system (e.g., Jira,
Azure DevOps) provided the system captures the required information.

Each problem report shall include:
a) Statement of criticality including:
   - Effect on performance
   - Effect on safety (reference to risk analysis)
   - Effect on security
b) Devices affected
c) Supported accessories affected
d) Software version(s) affected
e) Date and source of problem detection
f) Detailed problem description
g) Steps to reproduce (if applicable)

Problem reports shall be created for problems detected:
- During development (requirements, design, coding, integration, system testing)
- During verification and validation activities
- After release (complaints, field feedback, post-market surveillance)
- From internal sources (testing, code review, analysis)
- From external sources (users, customers, regulatory authorities)
```

**3. Add Notification to Relevant Parties (§9.3)**

**Recommended text for §6.13**:

```
6.13.3 Advise Relevant Parties

The manufacturer shall advise relevant parties of the existence of problems
as appropriate, based on the problem's criticality and safety impact.

Relevant parties may include:
- Internal: Development team, quality assurance, regulatory affairs,
  risk management, senior management
- External: Customers, users, regulatory authorities, notified bodies

For safety-related problems, notification procedures shall follow D0003325,
Global Procedure, Nonconformance, for complaint escalation and regulatory
reporting requirements.

The Software Development Plan or Software Maintenance Plan shall define
criteria for determining when notification is required and to which parties.
```

**4. Add Trend Analysis Requirements (§9.6)**

**Recommended text for §6.13**:

```
6.13.6 Trend Analysis

The manufacturer shall perform periodic analysis of problem reports to detect
trends that may indicate systemic quality or safety issues.

Trend analysis shall be conducted:
- At defined intervals (to be specified in Software Maintenance Plan)
- When a specified threshold of problem reports is reached
- In preparation for design reviews
- As part of post-market surveillance activities

Trend analysis shall examine:
- Problem frequency and severity over time
- Common root causes across multiple problem reports
- Problems affecting specific software components or modules
- Problems affecting specific device models or configurations
- Correlation with software versions and releases

When adverse trends are identified, the manufacturer shall:
a) Document the trend in the risk management file
b) Initiate corrective and preventive actions (CAPA) per D0003060
c) Update risk analysis as appropriate
d) Advise relevant parties per §6.13.3

Trend analysis results shall be documented and maintained as records per §6.13.5.
```

**5. Expand Verification Requirements (§9.7)**

**Recommended text for §6.13**:

```
6.13.7 Verify Problem Resolution

All problem resolutions shall be verified to ensure:
a) The problem has been resolved and the problem report can be closed
b) Adverse trends (if applicable) have been reversed
c) Change requests have been implemented in the appropriate medical device
   software and activities
d) Additional problems have not been introduced

Verification shall include:
- Re-testing the specific problem scenario
- Regression testing per §6.7 (integration) and §6.8 (system testing)
  as appropriate
- Review of test results against acceptance criteria
- Confirmation that changes followed the change control process per §6.12

For Class B and C devices, regression testing shall be performed to demonstrate
that the problem resolution has not introduced defects into previously tested
software.

Verification results shall be documented per §9.8 test documentation requirements.

Problem reports shall only be closed after:
- Verification is complete and successful
- All required records have been updated
- Relevant parties have been notified of resolution (as appropriate)
```

**6. Add Explicit Integration Points**
