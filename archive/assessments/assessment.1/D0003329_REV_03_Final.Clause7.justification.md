# Clause 7 Software Risk Management Process - Gap Justification Documentation

## Executive Summary

**Critical Gaps**: 9 identified
**Significant Gaps**: 10 identified
**Minor Gaps**: 4 identified

Clause 7 represents the **most significant compliance challenge** in D0003329 Rev 03. While §6.11 provides strong ISO 14971 integration and excellent §7.3.3 traceability requirements, it lacks critical **procedural detail** for executing §7.1 (hazard analysis), §7.3.1 (verification), and §7.4 (change risk management).

## Key Finding

D0003329 §6.11 reads as **POLICY** ("shall identify software items") rather than **PROCEDURE** ("how to identify software items"). This creates high regulatory risk as teams lack guidance for systematic execution.

## Critical Gaps

### Gap #1: Identify Potential Causes (§7.1.2)

#### IEC 62304 Requirement

> §7.1.2: "The MANUFACTURER shall identify potential causes of the SOFTWARE ITEM... contributing to a hazardous situation." Including five specific categories: (a) incorrect specification, (b) software defects, (c) SOUP failure, (d) hardware failures, (e) foreseeable misuse.

#### D0003329 Content

§6.11 Para 1: "identify software items that may contribute to a hazardous situation" (§7.1.1) but does NOT explicitly address §7.1.2 cause identification.

#### Gap Determination - **CRITICAL**

**How Determined**:

1. **Requirement Check**: IEC 62304 has distinct requirements:

   - §7.1.1: Identify software items
   - §7.1.2: Identify causes (separate requirement)

2. **D0003329 Coverage**:

   - §7.1.1: ✓ Addressed (Para 1)
   - §7.1.2: ❌ Not explicitly mentioned

3. **Five-Category Analysis**:
   - Checked D0003329 for each IEC 62304 §7.1.2(a-e) category
   - Result: NONE of the five categories are mentioned

**Why Critical**:

- **Incomplete Hazard Analysis**: Identifying software items WITHOUT identifying causes is incomplete
- **Standard Structure**: IEC 62304 deliberately separates item identification (§7.1.1) from cause identification (§7.1.2)
- **Safety Impact**: Causes drive risk control design - missing cause analysis means inadequate controls
- **Mandatory Requirement**: §7.1.2 uses "shall" - not optional

**Evidence of Gap**:

- D0003329 §6.11 Para 1 addresses §7.1.1 only
- No mention of "potential causes"
- No reference to five IEC 62304 categories
- No procedure for cause analysis

**Real-World Impact**:

- Team identifies: "Alarm software item contributes to hazard"
- Without §7.1.2: May not analyze WHY (incorrect alarm threshold? SOUP timer failure? Hardware sensor drift?)
- Result: Risk controls may not address root causes

**Recommended Fix**: Add explicit §7.1.2 procedure to §6.11 including all five cause categories with guidance.

### Gap #2: SOUP Anomaly List Evaluation (§7.1.3)

#### IEC 62304 Requirement

> §7.1.3: "EVALUATE as a minimum any ANOMALY list published by the supplier of the SOUP item relevant to the VERSION of the SOUP item used... to determine if any of the known ANOMALIES result in a sequence of events that could result in a hazardous situation."

#### D0003329 Content

§6.11 does not explicitly address SOUP anomaly list evaluation.

#### Gap Determination - **CRITICAL**

**How Determined**:

1. **Explicit Requirement Search**: Searched D0003329 §6.11 for "SOUP anomaly," "anomaly list," "published anomalies"

   - Result: Not found

2. **Contextual Coverage Check**: Reviewed §6.4 (Software Architecture, which addresses SOUP)

   - §6.4 requires SOUP functional requirements and hardware/software needed
   - §6.4 does NOT address anomaly evaluation

3. **Version-Specific Requirement**: IEC 62304 explicitly requires evaluation for "VERSION of the SOUP item used"
   - D0003329: No version-specific anomaly evaluation requirement

**Why Critical**:

- **Known Vulnerabilities**: SOUP suppliers publish known issues (CVEs, bug lists) - failure to evaluate means using SOUP with known safety-impacting defects
- **Version-Specific**: Different SOUP versions have different anomalies - must evaluate specific version in use
- **Regulatory Expectation**: FDA, EU regulators expect SOUP cybersecurity vulnerability assessment
- **Patient Safety**: Known SOUP anomalies (e.g., memory leaks, race conditions, security vulnerabilities) directly impact patient safety if not evaluated

**Evidence of Gap**:

- §6.11 does not mention SOUP anomaly evaluation
- §6.4 (SOUP architecture section) does not cross-reference anomaly evaluation
- No procedure for obtaining or evaluating SOUP supplier anomaly lists

**Real-World Example**:

- Device uses OpenSSL library version X
- OpenSSL publishes CVE (Common Vulnerabilities and Exposures) for version X
- Without §7.1.3: Team may not evaluate if CVE affects device
- Result: Cybersecurity vulnerability unaddressed

**Recommended Fix**: Add explicit SOUP anomaly evaluation requirement to §6.11 including procedure for obtaining and evaluating supplier-published anomaly lists for specific versions.

### Gap #3: Verify Risk Control Measures (§7.3.1)

#### IEC 62304 Requirement

> §7.3.1: "The implementation of each RISK CONTROL measure documented in 7.2 shall be VERIFIED, and this VERIFICATION shall be documented."

#### D0003329 Content

§6.11 Para 5 mentions "verification of risk control measure" in traceability context but does NOT explicitly require verification as a separate activity.

#### Gap Determination - **CRITICAL**

**How Determined**:

1. **Explicit Requirement Check**: IEC 62304 §7.3.1 has TWO mandatory requirements:

   - "shall be VERIFIED"
   - "VERIFICATION shall be documented"

2. **D0003329 Coverage Analysis**:

   - Para 5 (traceability): Mentions "verification of the risk control measure" as END POINT of traceability
   - Does NOT state: "Risk control measures shall be verified"
   - Does NOT state: "Verification shall be documented"

3. **Procedural Gap**: Even if traceability implies verification happened, §7.3.1 requires explicit verification requirement and documentation

**Why Critical**:

- **Unverified Controls**: Risk controls may be DESIGNED but not VERIFIED to work correctly
- **Safety Assumption**: Assuming controls work without verification creates safety risk
- **Mandatory Documentation**: IEC 62304 explicitly requires verification documentation - silence in §6.11 creates gap
- **Audit Evidence**: Auditors will ask for verification evidence - D0003329 doesn't require it

**Evidence of Gap**:

- §6.11 mentions verification in traceability context (Para 5)
- §6.11 does NOT state "risk control measures shall be verified" as requirement
- §6.11 does NOT state verification shall be documented
- No cross-reference to §6.8 (testing) for verification procedures

**Distinction**:

- **Traceability** (§7.3.3): Document the LINK from control to verification
- **Verification** (§7.3.1): Actually PERFORM and DOCUMENT the verification
- D0003329 addresses traceability but not verification requirement itself

**Recommended Fix**: Add explicit §7.3.1 requirement: "Risk control measures shall be verified. Verification shall be documented."

### Gap #5: Analyze Changes for New Hazard Causes (§7.4.1)

#### IEC 62304 Requirement

> §7.4.1: "The MANUFACTURER shall analyse changes to the MEDICAL DEVICE SOFTWARE (including SOUP) to determine whether: a) additional potential causes are introduced contributing to a hazardous situation"

#### D0003329 Content

§6.11 does not contain §7.4 (Risk Management of Software Changes) section.

#### Gap Determination - **CRITICAL**

**How Determined**:

1. **Section Presence Check**: IEC 62304 §7.4 is entire section (§7.4.1, §7.4.2, §7.4.3)

   - D0003329 §6.11: No §7.4 coverage

2. **Alternative Location Check**: Evaluated if §7.4 requirements might be in:

   - §6.10 (Maintenance): Addresses maintenance but not explicitly §7.4 risk analysis
   - §6.12 (Configuration Management): Addresses change control but not risk analysis
   - Result: §7.4 requirements not explicitly addressed anywhere

3. **Safety Class Analysis**: IEC 62304 §7.4.1 applies to ALL classes (A, B, C) - makes this especially critical

**Why Critical**:

- **Changes Introduce New Hazards**: Software changes are high-risk activities - analysis is mandatory
- **All Safety Classes**: §7.4.1 applies to Class A, B, C (unlike most §7 requirements which are B/C only)
- **Bypass Risk**: Without explicit requirement, changes may skip risk analysis
- **Integration Gap**: Maintenance (§6.10) and configuration management (§6.12) don't explicitly require §7.4 risk analysis

**Evidence of Gap**:

- §6.11 has no §7.4 section
- §6.10 (Maintenance) does not explicitly reference §7.4
- §6.12 (Configuration Management) does not explicitly reference §7.4
- No procedure for analyzing software changes for new hazard causes

**Impact**:

- **Safety**: Software changes may introduce unanalyzed hazards
- **Regulatory**: Major audit finding for missing mandatory risk analysis
- **Process**: Change control process incomplete without risk analysis

**Recommended Fix**: Add §7.4 Risk Management of Software Changes section to §6.11 with explicit requirements for analyzing changes per §7.4.1, §7.4.2, §7.4.3.
