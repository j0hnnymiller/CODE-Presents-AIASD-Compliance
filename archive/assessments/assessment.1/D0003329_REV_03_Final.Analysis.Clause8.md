## Executive Summary

D0003329 Rev 03 Section 6.12 provides **strong foundational compliance** with IEC 62304 Clause 8 requirements for software configuration management. The work instruction correctly addresses configuration planning integration, tool identification, configuration item control, SOUP management, repository requirements, and overall configuration management framework. However, **specific gaps exist in explicit procedural details** for configuration identification schemes (§8.1.1), SOUP documentation requirements (§8.1.2), system configuration documentation (§8.1.3), change approval workflows (§8.2.1), and configuration status accounting (§8.3).

**Overall Compliance Status**:

- **Compliant Requirements**: 6 of 11 (55%)
- **Partial Compliance**: 5 of 11 (45%)
- **Non-Compliant/Missing**: 0 of 11 (0%)

**Key Strengths**:

- D0003329 §6.12 correctly requires a Software Configuration Management Plan with comprehensive scope
- Strong integration with Software Development Planning (§6.2) and maintenance processes (§6.10)
- Excellent repository control and archival requirements
- Good coverage of configuration item categories and SOUP handling
- Tool validation requirements for Class B/C devices properly addressed

**Critical Finding**: While D0003329 §6.12 establishes the _requirement_ for configuration management planning and identifies key elements, it lacks explicit _procedural detail_ for executing specific IEC 62304 Clause 8 requirements. The section delegates heavily to the Software Configuration Management Plan (D0004170) without stating minimum mandatory content for plan compliance.

### 8.1.2: Identify SOUP

**IEC 62304 §8.1.2 Requirement**:

> "For each SOUP CONFIGURATION ITEM being used, including standard libraries, the MANUFACTURER shall document:
> a) the title,
> b) the MANUFACTURER, and
> c) the unique SOUP designator
> [Class A, B, C]
> NOTE: The unique SOUP designator could be, for example, a VERSION, a release date, a patch number or an upgrade designation."

#### Partial Compliance ⚠️

| Element                           | D0003329 Reference       | Gap Description                                                                                                                                                                                                                      |
| --------------------------------- | ------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| SOUP identification               | §6.12, Para 2 (bullet 2) | "A description of how the software is controlled, including how changes are managed, and how SOUP is handled" mentions SOUP but doesn't explicitly state the three required documentation elements (title, manufacturer, designator) |
| Title requirement (a)             | Not explicit             | Not explicitly stated as mandatory SOUP documentation element                                                                                                                                                                        |
| Manufacturer requirement (b)      | Not explicit             | Not explicitly stated as mandatory SOUP documentation element                                                                                                                                                                        |
| Unique designator requirement (c) | Not explicit             | Not explicitly stated as mandatory SOUP documentation element                                                                                                                                                                        |
| Standard libraries                | Not explicit             | "including standard libraries" not mentioned                                                                                                                                                                                         |

#### Observations 📋

**Gap Assessment**: D0003329 §6.12 states that SOUP shall be "handled" in the Configuration Management Plan but does not enumerate the three specific IEC 62304 §8.1.2 documentation requirements (title, manufacturer, unique designator). This is a **significant gap** as SOUP identification is critical for reproducibility and change management.

**Integration Note**: D0003329 §6.4 (Software Architecture) addresses SOUP description requirements including "functional and performance requirements" and "hardware and software necessary to support the SOUP," but §8.1.2-specific identification elements (title, manufacturer, designator) are not explicitly cross-referenced in §6.12.

## Section 8.2: Change Control

**IEC 62304 §8.2 Context**:

> Change control ensures that changes to configuration items are approved, implemented per plans, verified, and traceable.

### 8.2.1: Approve Change Requests

**IEC 62304 §8.2.1 Requirement**:

> "The MANUFACTURER shall change CONFIGURATION ITEMS identified to be controlled according to 8.1 only in response to an approved CHANGE REQUEST. [Class A, B, C]
> NOTE 1: The decision to approve a CHANGE REQUEST can be integral to the change control PROCESS or part of another PROCESS. This subclause only requires that approval of a change precede its implementation.
> NOTE 2: Different acceptance PROCESSES can be used for CHANGE REQUESTS at different stages of the life cycle, as stated in plans, see 5.1.1 d) and 6.1 e)."

#### Compliant Elements ✅

| Element                        | D0003329 Reference       | Evidence                                                                             |

## Configuration Management Plan (D0004170) Assessment

**Context**: D0003329 §6.12 requires use of D0004170 (Global Template, Software Configuration Management Plan) but doesn't specify minimum mandatory content for plan compliance.

### Coverage Analysis

**What D0003329 §6.12 Requires in Plan**:

1. Tools used in developing the software (e.g., compilers, bug trackers)
2. A description of how the software is controlled, including how changes are managed, and how SOUP is handled
3. Categories (or lists) of items that will be controlled
4. Tasks and activities related to Configuration Management
5. [Class B/C] How items that support software development are controlled (tool validation)

**What IEC 62304 Clause 8 Requires** (implied content for plan):

1. Scheme for unique identification of configuration items and versions (§8.1.1)
2. SOUP documentation requirements: title, manufacturer, designator (§8.1.2)
3. System configuration documentation approach (§8.1.3)
4. Change request approval process (§8.2.1)
5. Change implementation procedures (§8.2.2)
6. Change verification procedures (§8.2.3)
7. Change traceability mechanisms (§8.2.4)
8. Configuration status accounting/history records (§8.3)

### Gap Assessment ⚠️

**Gaps in D0003329 §6.12 Specification of Plan Content**:

- Identification scheme (§8.1.1) - mentioned as "categories" but not explicit naming/versioning scheme requirement
- SOUP documentation elements (§8.1.2) - not enumerated
- System configuration documentation (§8.1.3) - not explicit
- Change approval workflow (§8.2.1) - not explicit
- Safety classification review in changes (§8.2.2) - not mentioned
- Verification procedures for changes (§8.2.3) - not explicit in §6.12
- Three-way traceability (§8.2.4) - not explicit
- Status accounting (§8.3) - addressed via repository requirement but not as explicit plan element

**Risk**: If D0004170 template doesn't include these elements, projects may create configuration management plans that don't fully comply with IEC 62304 Clause 8.

**Recommendation**: Either (1) expand D0003329 §6.12 to explicitly enumerate all required plan elements per §8.1-8.3, or (2) verify D0004170 template includes all elements and add statement in §6.12: "The Configuration Management Plan shall address all requirements of IEC 62304 Clause 8 (§8.1-8.3)"

### Priority 1: Significant Gaps (Address in Next Revision Cycle - Weeks 1-4)

#### Recommendation 1.1: Add Explicit SOUP Documentation Requirements (§8.1.2)

**Action**: Expand §6.12 to explicitly require the three SOUP documentation elements per IEC 62304 §8.1.2.

**Proposed Addition to D0003329 §6.12** (new paragraph after Para 5, or integrate into Para 2):

```
SOUP Identification (IEC 62304 §8.1.2)

For each SOUP configuration item being used, including standard libraries, the following shall
be documented:

a) The title (name of the SOUP component)
b) The manufacturer or supplier
c) The unique SOUP designator (version number, release date, patch number, or upgrade
   designation)

SOUP identification shall be maintained in the Software Configuration Management Plan,
Software Architecture (§6.4), or controlled repository records. The identification shall
be sufficient to uniquely identify the specific SOUP version used in the medical device.

Cross-reference: See §6.4 (Software Architecture) for SOUP functional and performance
requirements.
```

**Standard Citation**: IEC 62304 §8.1.2

#### Recommendation 1.2: Add Safety Classification Review Requirement for Changes (§8.2.2)

**Action**: Explicitly require review of software safety classification when changes are made.

**Proposed Addition to D0003329 §6.12** (new bullet in Para 2 or new paragraph):

```
Safety Classification Review for Changes

Changes to software configuration items shall be evaluated to determine whether the change
affects the software safety classification of the SOFTWARE SYSTEM or SOFTWARE ITEMS.
If the change impacts:
- Hazardous situations that the software can contribute to,
- Risk control measures implemented in software, or
- Software functionality related to patient safety,

the software safety classification shall be reviewed and updated as appropriate per §6.11
(Software Risk Management) and documented in the Risk Management File per D0003103.

This review shall be performed as part of the change implementation process and documented
in the change request or modification record.

Cross-reference: See §6.11 for software safety classification requirements and Appendix 10.1
for Software Safety Classification Process.
```

**Standard Citation**: IEC 62304 §8.2.2 ("including changes to the software safety classification")
**Estimated Effort**: 2 hours
**Verification**: Integration check with §6.11, §7.4 (risk management of changes), and change control workflow

#### Recommendation 1.3: Add Three-Way Traceability Requirement (§8.2.4)

**Action**: Explicitly require traceability between change requests, problem reports, and approvals.

**Proposed Addition to D0003329 §6.12** (new paragraph after change control discussion):

```
Change Traceability (IEC 62304 §8.2.4)

Records shall be maintained of the relationships and dependencies between:
a) Change requests
b) Relevant problem reports (anomalies, complaints, or other issues)
c) Approval of the change request

This traceability shall enable determination of:
- Which problem reports resulted in which change requests
- Whether change requests were approved and by whom
- Which changes were implemented in response to specific problems
- The complete history of a configuration item change from problem identification through
  approval and implementation

Traceability may be maintained through:
- Configuration management tool features (e.g., linking change requests to problem reports)
- Change request documentation that references problem report IDs
- Approval records that reference change request IDs
- Traceability matrix linking problem reports, change requests, and approvals

Cross-reference: See §6.13 (Software Problem Resolution Process) for problem report tracking
requirements.
```

**Standard Citation**: IEC 62304 §8.2.4
**Estimated Effort**: 2-3 hours
**Verification**: Integration validation with §6.13 and problem tracking tools (Jira, Azure DevOps, etc.)

#### Recommendation 1.5: Add System Configuration Documentation Requirement (§8.1.3)

**Action**: Explicitly require documentation of the set of configuration items and versions comprising each software system release.

**Proposed Addition to D0003329 §6.12** (new paragraph after Para 5):

```
System Configuration Documentation (IEC 62304 §8.1.3)

The set of configuration items and their versions that comprise each SOFTWARE SYSTEM
configuration shall be documented. This system configuration documentation shall:

a) Identify all configuration items included in the software system release (source code
   modules, executables, documentation, SOUP components, build scripts, configuration files)
b) Specify the version of each configuration item included in the release
c) Uniquely identify the software system configuration (e.g., software version number,
   build number, release identifier)
d) Enable recreation of the exact software system configuration if needed

System configuration documentation may be implemented through:
- Version Description Document (D0004199, Global Template, Version Description Document)
- Software release bill of materials
- Configuration management tool reports (e.g., repository tags, build manifests)
- Software release records maintained per §6.9

Cross-reference: See §6.9 (Software Release) for release documentation requirements.
```

**Standard Citation**: IEC 62304 §8.1.3

## Regulatory Risk Assessment

### Impact of Non-Compliance

**Moderate Risk** (Significant Gaps):

- **SOUP Traceability**: Missing explicit SOUP documentation requirements (§8.1.2) could result in inability to reproduce software or assess impact of SOUP vulnerabilities/obsolescence
- **Safety Classification Drift**: Failure to review safety classification during changes (§8.2.2) could result in inadequate lifecycle rigor for software that has become safety-critical
- **Change Traceability**: Missing three-way traceability (§8.2.4) creates difficulty demonstrating that problems were properly resolved and changes were authorized
- **Change Control**: Lack of explicit approval gate (§8.2.1) could allow unauthorized changes, increasing quality risk
- **System Configuration**: Missing system configuration documentation (§8.1.3) impairs ability to reproduce specific software releases for investigation or updates
- **Regulatory Submission**: Reviewers expect explicit IEC 62304 Clause 8 compliance; gaps may result in additional questions or deficiency findings
- **Audit Findings**: Missing configuration management elements likely to result in observations or minor audit findings

**Low Risk** (Moderate/Minor Gaps):

- **Procedural Clarity**: Lack of detailed procedures reduces consistency but doesn't prevent compliance if practices are sound
- **Cross-Reference Gaps**: Missing cross-references create inefficiency but don't prevent compliance
- **Terminology**: Using different terminology (e.g., "repository" vs. "configuration status accounting") is acceptable if intent is met

## Conclusion

D0003329 Rev 03 Section 6.12 provides a **strong foundational framework** for IEC 62304 Clause 8 compliance with particularly excellent coverage of repository control, tool validation (Class B/C), and overall configuration management planning. The work instruction correctly establishes the requirement for a Software Configuration Management Plan and identifies key elements that must be addressed. However, **specific procedural gaps** exist in SOUP documentation requirements (§8.1.2), safety classification review during changes (§8.2.2), three-way change traceability (§8.2.4), and explicit approval gates (§8.2.1) that create moderate compliance risk.

### Key Takeaways

1. **Strong Foundation**: §6.12 establishes comprehensive configuration management framework with plan requirement and key elements
2. **Exemplary Repository Control**: Repository requirements (validated, controlled, auditable, lifecycle retention) exceed minimum IEC 62304 expectations
3. **Excellent Tool Validation**: Class B/C tool validation requirement with D0003078 cross-reference is best-in-class
4. **Delegation to Plan**: §6.12 delegates heavily to Configuration Management Plan without fully specifying minimum plan content for §8.1-8.3 compliance
5. **SOUP Gap**: §8.1.2 three-element SOUP documentation requirement (title, manufacturer, designator) not explicit
6. **Safety Classification Integration**: Critical §8.2.2 requirement to review safety classification during changes not addressed
7. **Traceability Gap**: Three-way traceability (change request ↔ problem report ↔ approval) per §8.2.4 not explicit
8. **Change Approval**: Explicit approval gate per §8.2.1 not stated
9. **No Critical Gaps**: All gaps are procedural/documentation gaps that don't directly impact patient safety

### Recommended Next Steps

1. **Immediate Action** (Priority 1 - Weeks 1-4): Implement significant gap remediation (SOUP documentation, safety classification review, three-way traceability, change approval, system configuration documentation)
2. **Short-Term** (Priority 2 - Weeks 5-8): Enhance procedural detail, strengthen cross-references, update D0004170 Configuration Management Plan Template
3. **Continuous Improvement** (Priority 3 - Weeks 9-12): Add examples, guidance, and best practices; pilot revised procedures
4. **Template Alignment**: Verify D0004170 (Configuration Management Plan Template) includes all elements to support §8.1-8.3 compliance
5. **Cross-Section Integration**: Strengthen linkages between §6.12 and §6.2, §6.8, §6.10, §6.11, §6.13 for seamless configuration management execution

### Compliance Confidence Level

**Current State**: **Good Partial Compliance** - Strong framework with procedural gaps
**Post-Remediation (Priority 1)**: **Strong Compliance** - Significant gaps closed with explicit requirements
**Post-Remediation (All)**: **Exemplary** - Comprehensive configuration management with detailed procedures and industry best practices
