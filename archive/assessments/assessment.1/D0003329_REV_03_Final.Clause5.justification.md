## Gap Determination Methodology

### Classification Criteria for Partial Compliance (⚠️)

A requirement was classified as "Partial Compliance" when:

1. **Implicit vs. Explicit**: D0003329 addresses the requirement implicitly (through references, related sections, or general statements) but does not explicitly state the requirement in clear, direct language
2. **High-Level vs. Detailed**: D0003329 provides high-level coverage without procedural detail on HOW to execute the requirement
3. **Template Dependency**: D0003329 relies on templates (e.g., D0004169, D0017985) to contain specific content without explicitly stating what that content must include in the work instruction
4. **Incomplete Coverage**: D0003329 addresses part of a multi-element requirement but does not cover all elements
5. **Integration Gaps**: D0003329 provides isolated coverage without explicit integration or cross-referencing to related processes

## Section 5.1: Software Development Planning - Partial Compliance Items

### Gap #1: Processes to Be Used (§5.1.1.a)

#### IEC 62304 Requirement

> §5.1.1.a: "a) the PROCESSES to be used in software development;"

#### D0003329 Content Reviewed

**§6.2**: "A Software Development Plan shall be developed that may use D0004168, Global Template, Software Development Plan, or equivalent. The plan shall address the following: Identification of the processes to be used"

Also references: "Per applicable requirements of BS EN 62304... [and] applicable SOPs"

#### Gap Determination

**Status**: ⚠️ **PARTIAL COMPLIANCE**

**How Determined**:

1. **Explicit Statement Analysis**:

   - IEC 62304 requires: "the PROCESSES to be used in software development"
   - D0003329 requires: "Identification of the processes to be used"
   - ✓ Correctly addresses requirement to identify processes

2. **Specificity Analysis**:

   - IEC 62304 implies the plan should **describe** which processes will be used (e.g., waterfall, iterative, agile)
   - D0003329 states "identification of processes" but does not require **description** of how processes will be implemented
   - Gap: Teams may list process names without describing their application

3. **Integration Assessment**:
   - D0003329 references "BS EN 62304" and "applicable SOPs" but does not clearly state which processes from these sources apply
   - Unclear: Does organization use all IEC 62304 processes? Are any tailored? Which SOPs implement which processes?

**Evidence of Partial Coverage**:

- D0003329 correctly requires "identification" ✓
- D0003329 does not require description of how processes are implemented ⚠️
- D0003329 does not clarify whether waterfall, iterative, or agile approach is used ⚠️
- D0003329 references standards/SOPs but doesn't map process implementation approach ⚠️

**Why Partial (Not Non-Compliant)**:

- The requirement IS addressed - "identification of processes" satisfies basic §5.1.1.a requirement
- However, regulatory best practice and IEC 62304 intent suggest plans should describe HOW processes are used, not just list them
- This is an enhancement opportunity rather than a critical gap

**Regulatory Basis for Gap**:

- **FDA Guidance**: Software development plans should clearly describe the development approach (waterfall, agile, iterative hybrid) and how IEC 62304 processes map to the approach
- **Auditor Expectations**: Auditors typically expect plans to describe the development model and process flow, not just list process names
- **IEC 62304 Annex B**: Guidance materials suggest plans should provide enough detail to understand the software development approach

**Impact of Gap**:

- **Minor**: Teams can develop compliant plans, but lack of explicit guidance may result in inconsistent plan quality (some detailed, some just listing processes)
- **Clarity**: Plans may not clearly communicate the development approach to stakeholders
- **Audit**: Minor observation possible if plans lack process description

**Recommended Enhancement**:

Clarify D0003329 §6.2:

```
Identification of the processes to be used in software development, including:
- Description of the software development approach (e.g., waterfall, iterative, agile)
- How BS EN 62304 processes map to the selected development approach
- Which organizational SOPs implement which IEC 62304 processes
```

### Gap #2: Traceability to Risk Control Measures (§5.1.1.c)

#### IEC 62304 Requirement

> §5.1.1.c: "c) software configuration management ACTIVITIES (see Clause 8);"
> [Later reference: §5.1.1 context includes traceability]
> §5.1.6 and §5.2.3 context: Traceability should include risk control measures

#### D0003329 Content Reviewed

**§6.2**: "Traceability of Software Requirements, Design, Test, and risk controls using D0017985, Global Template, Software Traceability Matrix, or equivalent"

#### Gap Determination

**Status**: ⚠️ **PARTIAL COMPLIANCE**

**How Determined**:

1. **Explicit Language Comparison**:

   - IEC 62304 context from multiple sections (§5.1.6, §5.2.3, §7.3.3): Traceability should include risk control measures implemented in software
   - D0003329 states: "traceability of... risk controls"
   - ✓ Mentions risk controls

2. **Scope Clarity Analysis**:

   - D0003329 uses term "risk controls" (general)
   - IEC 62304 uses "RISK CONTROL measures implemented in software" (specific)
   - Gap: Unclear if traceability scope includes ALL risk controls or only those implemented in software

3. **Template Dependency**:
   - D0003329 references D0017985 "Software Traceability Matrix" for implementation
   - Gap: D0003329 does not explicitly state what risk control traceability must include (control → requirement → design → test → verification)
   - Risk: If D0017985 template is inadequate or not used, risk control traceability may be incomplete

**Evidence of Partial Coverage**:

- D0003329 mentions "risk controls" in traceability requirement ✓
- D0003329 does not clarify "risk control measures implemented in software" (IEC 62304 language) ⚠️
- D0003329 does not specify the complete risk control traceability chain ⚠️
- D0003329 delegates details to template without stating minimum content ⚠️

**Why Partial (Not Non-Compliant)**:

- Risk controls ARE mentioned in traceability requirement
- However, the scope and specific linkages required by IEC 62304 could be more explicit
- This is primarily a clarity/specificity issue rather than missing requirement

**Regulatory Basis for Gap**:

- **IEC 62304 §7.3.3**: Explicitly requires four-link traceability chain for risk control measures (hazard → software item → cause → control → verification)
- **Integration**: Traceability planning (§5.1) should explicitly integrate with risk management traceability requirements (§7.3.3)
- **Audit Expectations**: Auditors expect clear demonstration that software-implemented risk controls are traceable through requirements → design → implementation → verification

**Impact of Gap**:

- **Minor**: Risk control traceability IS required by D0003329, but lack of specificity may lead to incomplete implementation
- **Template Dependency Risk**: If D0017985 template does not fully implement risk control traceability per §7.3.3, gap may not be caught
- **Integration**: Weak linkage between §6.2 (planning) and §6.11 (risk management process that requires traceability per §7.3.3)

**Recommended Enhancement**:

Clarify D0003329 §6.2:

```
Traceability of Software Requirements, Design, Test, and risk controls including:
- Requirements traceability: System requirements → Software requirements → Design → Implementation → Tests
- Risk control traceability: Risk control measures implemented in software shall be traceable
  through requirements → design → implementation → verification (per §6.11 and IEC 62304 §7.3.3)
- Template: D0017985, Global Template, Software Traceability Matrix, or equivalent
```

### Gap #3: System Integration Procedures (§5.1.3.b)

#### IEC 62304 Requirement

> §5.1.3.b: "b) the procedures for coordinating the development of the MEDICAL DEVICE SOFTWARE with the development and testing of the MEDICAL DEVICE SYSTEM, including ACTIVITIES such as integration, VERIFICATION and VALIDATION activities;"

#### D0003329 Content Reviewed

**§6.2**: Does not explicitly address system integration coordination procedures. Verification is addressed generally but not system-level coordination.

**§6.7** (Integration): Addresses software integration but not explicitly system integration

#### Gap Determination

**Status**: ⚠️ **PARTIAL COMPLIANCE**

**How Determined**:

1. **Requirement Scope**:

   - IEC 62304 §5.1.3.b is about SOFTWARE-to-SYSTEM coordination (not software-to-software integration)
   - "coordinating... MEDICAL DEVICE SOFTWARE with... MEDICAL DEVICE SYSTEM" = how software development interfaces with system-level activities

2. **D0003329 Coverage Analysis**:

   - §6.7 addresses software integration and integration testing (software items integrating)
   - §6.8 addresses software system testing (testing complete software system)
   - Missing: Explicit procedures for how software development coordinates with system-level activities

3. **Real-World Scenario**:
   - Medical device system may include hardware, mechanical, electrical, and software subsystems
   - Software team needs procedures for:
     - When/how software is delivered to system integration
     - How software verification interfaces with system validation
     - Who coordinates integration activities between subsystems
   - D0003329: Does not explicitly address these coordination procedures

**Evidence of Partial Coverage**:

- D0003329 §6.7 addresses software integration ✓
- D0003329 §6.8 addresses software system testing ✓
- D0003329 does not explicitly address software-to-system coordination procedures ⚠️
- D0003329 does not clarify how software verification relates to system validation ⚠️

**Why Partial (Not Non-Compliant)**:

- Integration and verification ARE addressed in D0003329
- However, explicit procedures for coordinating software with SYSTEM development are not clearly stated
- This may be addressed in system-level procedures (outside D0003329 scope) but IEC 62304 requires SOFTWARE development plan to address this

**Regulatory Basis for Gap**:

- **IEC 62304 Intent**: Software development doesn't occur in isolation - coordination with system engineering is critical
- **IEC 60601-1 Integration**: Medical electrical equipment standard requires system-level validation; IEC 62304 requires software plan to describe coordination
- **FDA Guidance**: Software development plans should describe integration touchpoints with system development

**Impact of Gap**:

- **Minor to Moderate**: Coordination likely happens in practice but lack of explicit procedures in D0003329 creates risk of inconsistent coordination
- **System Integration Issues**: Without explicit coordination procedures, software-hardware integration problems may occur
- **Validation Gaps**: Unclear boundaries between software verification and system validation

**Recommended Enhancement**:

Add to D0003329 §6.2:

```
Procedures for coordinating software development with system development, including:
- Integration of software with device system hardware and other subsystems
- Coordination between software verification activities (§6.8) and system validation activities
- Interfaces and deliverables between software team and system engineering team
- Responsibilities for system integration activities
```

### Gap #4: Software Configuration Item Control Before Verification (§5.1.11)

#### IEC 62304 Requirement

> §5.1.11: "The plan shall state when each CONFIGURATION ITEM is to be placed under the control of the configuration management PROCESS, but it shall be BEFORE the CONFIGURATION ITEM is VERIFIED or RELEASED to production." [Class B, C]

#### D0003329 Content Reviewed

**§6.12** (Configuration Management): "The Software Configuration Management Plan shall describe... when the items are to be placed under configuration control"

Does NOT explicitly state "BEFORE verification or release"

#### Gap Determination

**Status**: ⚠️ **PARTIAL COMPLIANCE**

**How Determined**:

1. **Timing Requirement**:

   - IEC 62304 explicitly requires: Configuration items must be under control **BEFORE verification**
   - D0003329 requires: Plan shall describe "when" items are placed under control
   - Gap: D0003329 does not explicitly state the "BEFORE verification" timing constraint

2. **Critical Timing Analysis**:

   - IEC 62304 rationale: Configuration items must be frozen/controlled before verification to ensure what is verified is what is released
   - If configuration item is not controlled before verification, verification results may not apply to final release configuration
   - D0003329 delegates timing decision to plan without stating the mandatory constraint

3. **Audit Scenario**:
   - Auditor: "When are configuration items placed under control?"
   - D0003329 answer: "Per the Software Configuration Management Plan"
   - Auditor: "Does the plan ensure control happens before verification?"
   - D0003329: Silent - no explicit requirement

**Evidence of Partial Coverage**:

- D0003329 requires plan to state "when" items are controlled ✓
- D0003329 does not state the mandatory "BEFORE verification" constraint ⚠️
- D0003329 does not state the mandatory "BEFORE release" constraint ⚠️
- Risk: Plans may specify control timing that violates IEC 62304 requirement

**Why Partial (Not Non-Compliant)**:

- D0003329 DOES require plans to address timing
- However, lack of explicit constraint means teams may not realize "before verification" is mandatory
- This is a specification gap rather than complete omission

**Regulatory Basis for Gap**:

- **IEC 62304 Explicit Language**: "but it shall be BEFORE..." uses mandatory "shall" language - this is not optional
- **Verification Integrity**: Fundamental principle - can only verify what is configuration-controlled
- **Class B/C Requirement**: This is safety-class-specific requirement that D0003329 does not explicitly scope

**Impact of Gap**:

- **Moderate**: Without explicit "before verification" constraint, teams may develop plans that allow verification of uncontrolled items
- **Verification Validity**: Verification results may not apply to released configuration if items change after verification
- **Audit Risk**: Auditor may identify that configuration management timing does not meet IEC 62304 requirement

**Recommended Enhancement**:

Revise D0003329 §6.12:

```
The Software Configuration Management Plan shall describe... when the items are to be placed
under configuration control. For Class B and C devices, configuration items shall be placed
under control BEFORE verification or release to production per IEC 62304 §5.1.11.
```

## Section 5.2: Software Requirements Analysis - Partial Compliance Items

### Gap #5: Security Requirements Explicit Statement (§5.2.2.e)

#### IEC 62304 Requirement

> §5.2.2: "Software requirements include... e) security, including those related to the compromise of clinically sensitive data;"

#### D0003329 Content Reviewed

**§6.3**: References D0004169 template (which is expected to include security)

**§6.11**: Addresses cybersecurity risk management with reference to D0029257

#### Gap Determination

**Status**: ⚠️ **PARTIAL COMPLIANCE**

**How Determined**:

1. **Explicit Requirement Analysis**:

   - IEC 62304 §5.2.2.e: Security requirements must be included in software requirements
   - D0003329 §6.3: References D0004169 template, does not explicitly state security requirements must be included
   - Gap: Requirement is implicit through template, not explicit in work instruction

2. **Integration Analysis**:

   - D0003329 §6.11 addresses cybersecurity (D0029257 reference)
   - However, §6.3 (requirements section) does not explicitly cross-reference security or cybersecurity
   - Gap: Weak integration between security risk management (§6.11) and requirements definition (§6.3)

3. **Comprehensiveness**:
   - IEC 62304 specifically mentions "compromise of clinically sensitive data" as example
   - D0003329 §6.11 addresses cybersecurity risk management
   - Gap: §6.3 does not explicitly require security requirements to address clinically sensitive data

**Evidence of Partial Coverage**:

- D0003329 §6.11 addresses cybersecurity risk management comprehensively ✓
- D0003329 §6.3 references D0004169 template (may include security) ✓
- D0003329 §6.3 does not explicitly state security requirements must be included ⚠️
- D0003329 §6.3 does not cross-reference §6.11 for security requirements ⚠️
- D0003329 §6.3 does not mention clinically sensitive data ⚠️

**Why Partial (Not Non-Compliant)**:

- Security IS addressed in D0003329 (§6.11)
- Requirements template (D0004169) likely includes security section
- However, explicit statement in requirements section (§6.3) would strengthen compliance and ensure integration

**Regulatory Basis for Gap**:

- **IEC 62304 §5.2.2.e**: Security requirements are mandatory part of software requirements specification
- **Cybersecurity Emphasis**: FDA, EU MDR, and global regulators emphasize cybersecurity; explicit security requirements are expected
- **Data Protection**: HIPAA, GDPR, and medical device regulations require protection of patient/clinical data

**Impact of Gap**:

- **Minor**: Security IS addressed through cybersecurity risk management (§6.11) and likely in template (D0004169)
- **Integration Risk**: Developers may not explicitly consider security when writing software requirements if §6.3 doesn't mention it
- **Audit**: Minor observation possible for lack of explicit security requirements statement in §6.3

**Recommended Enhancement**:

Add to D0003329 §6.3:

```
Software Requirements shall include security requirements, including those related to:
- Protection of clinically sensitive data
- Cybersecurity controls per §6.11 and D0029257
- Data integrity and confidentiality
- Authentication and access control
```

### Gap #6: Risk Analysis Re-evaluation Timing (§5.2.4)

#### IEC 62304 Requirement

> §5.2.4: "The MANUFACTURER shall re-EVALUATE the MEDICAL DEVICE RISK ANALYSIS upon the establishment of the software requirements."

#### D0003329 Content Reviewed

**§6.11**: "Risk analysis of products shall be in accordance with the overall project Risk Management Plan. This shall identify software items that may contribute to a hazardous situation..."

Describes continuous risk management but not explicit trigger tied to requirements establishment.

#### Gap Determination

**Status**: ⚠️ **PARTIAL COMPLIANCE**

**How Determined**:

1. **Trigger Analysis**:

   - IEC 62304 specifies explicit trigger: "**upon the establishment** of the software requirements"
   - D0003329 describes ongoing risk management but does not specify timing trigger
   - Gap: When should risk analysis be updated? IEC 62304 says: when requirements are established. D0003329: Unclear.

2. **Lifecycle Integration**:

   - IEC 62304 ties risk management activities to specific lifecycle phases (requirements → re-evaluate risk analysis)
   - D0003329 §6.11 describes general risk management process without explicit lifecycle phase triggers
   - Gap: Missing explicit statement that risk analysis SHALL be updated when requirements are established

3. **Cross-Reference Analysis**:
   - D0003329 §6.3 (Requirements) does not reference §6.11 (Risk Management) for re-evaluation
   - D0003329 §6.11 (Risk Management) does not reference §6.3 for requirements-established trigger
   - Gap: Weak integration between requirements analysis and risk management update trigger

**Evidence of Partial Coverage**:

- D0003329 §6.11 describes comprehensive risk management process ✓
- D0003329 describes risk management as continuous/integrated ✓
- D0003329 does not explicitly state trigger "when software requirements are established" ⚠️
- D0003329 does not cross-reference risk management update from requirements section ⚠️

**Why Partial (Not Non-Compliant)**:

- Risk management process IS described and is intended to be continuous
- However, IEC 62304 specifies a mandatory trigger point - lack of explicit trigger may result in teams not updating risk analysis at the right time

**Regulatory Basis for Gap**:

- **IEC 62304 Intent**: Requirements establishment is key milestone - risk analysis must be updated to reflect new understanding of system hazards based on defined requirements
- **ISO 14971 Integration**: Risk management is continuous, but key lifecycle milestones should trigger explicit risk re-evaluation
- **Audit Trail**: Auditors expect to see evidence of risk analysis updates tied to requirements phase

**Impact of Gap**:

- **Minor**: Risk management is ongoing, so updates likely happen, but lack of explicit trigger may cause delays or omissions
- **Process Maturity**: Explicit triggers strengthen process maturity and ensure systematic risk management
- **Documentation**: Risk management file may lack clear evidence of risk re-evaluation tied to requirements establishment

**Recommended Enhancement**:

Add to D0003329 §6.11:

```
Risk analysis shall be re-evaluated at key lifecycle milestones including:
- Upon establishment of software requirements (§6.3 / IEC 62304 §5.2.4)
- Upon completion of software architecture (§6.4 / IEC 62304 §5.3)
- Upon identification of software changes (§6.10 / IEC 62304 §7.4)
- Throughout development as design evolves
```
