### 6.2.1.1 Monitor Feedback

**Standard Requirement**: The manufacturer shall monitor feedback on medical device software released for intended use. [Class A, B, C]

**SOP Implementation** (§6.10):
The SOP requires "procedures for receiving, documenting, evaluating, resolving, and tracking feedback" within the maintenance plan but does not explicitly mandate continuous monitoring activities or specify monitoring mechanisms.

**Compliance Rating**: ⚠️ **PARTIAL COMPLIANCE (70%)**

**Analysis**: While the SOP addresses feedback _receipt_ and _documentation_, it lacks explicit requirements for proactive _monitoring_ as distinct from reactive receipt. IEC 62304:2006 §6.2.1.1 emphasizes ongoing surveillance of released software, which implies active collection from multiple sources (user reports, field observations, complaint systems, post-market surveillance). The SOP's integration with complaint management partially addresses this through reactive channels, but the absence of "monitoring" terminology and specification of monitoring methods creates ambiguity about proactive surveillance obligations.

**Gaps**:

- No explicit "monitoring" requirement stated
- Absence of monitoring mechanism specifications (e.g., surveillance systems, feedback channels)
- Limited guidance on proactive vs. reactive feedback collection
- No frequency or scope requirements for monitoring activities

**Recommendation**: Add explicit monitoring requirement: "The Software Maintenance Plan shall establish procedures for continuous monitoring of feedback on released medical device software. Monitoring mechanisms shall include, as applicable: complaint systems, user feedback channels, post-market surveillance data, field service reports, and regulatory notifications."

### 6.2.1.2 Document and Evaluate Feedback

The SOP requires "procedures for receiving, documenting, evaluating, resolving, and tracking feedback" and specifies "criteria for determining whether feedback is considered a problem" with reference to Requirements for Complaint Management. **Compliance Rating**: ✅ **SUBSTANTIAL COMPLIANCE (90%)**

**Strengths**:

- Clear documentation and evaluation requirements
- Problem determination criteria explicitly required
- Integration with problem resolution process (§6.13)
- Cross-reference to complaint management procedure

**Gaps**:

- Does not explicitly state "problem report" terminology in §6.10
- Content requirements for problem reports (adverse events, specification deviations) not stated in maintenance section

**Recommendation**: Add explicit language: "Any problem shall be recorded as a Problem Report in accordance with Section 6.13 and shall include actual or potential adverse events and deviations from specifications."

### 6.2.1.3 Evaluate Problem Report's Effects on Safety

**Standard Requirement**: Each problem report shall be evaluated to determine how it affects the safety of medical device software released for intended use (see 9.2) and whether a change to that software is needed to address the problem. [Class A, B, C]

**SOP Implementation** (§6.10, §6.13):
Section 6.10 requires "use of the software risk management process" within the maintenance plan. Section 6.13 (Software Problem Resolution Process) states "all anomalies identified shall be investigated, evaluated for safety, and will result in a change request or rationale for taking no action."

**Compliance Rating**: ✅ **SUBSTANTIAL COMPLIANCE (85%)**

**Analysis**: The SOP addresses safety evaluation through two mechanisms: (1) risk management process integration in §6.10, and (2) explicit safety evaluation requirement in §6.13. This dual approach ensures that safety considerations are embedded in both maintenance planning and problem resolution execution. The §6.13 requirement to evaluate anomalies "for safety" and determine whether change is needed directly satisfies IEC 62304:2006 §6.2.1.3. However, the connection between §6.10 maintenance planning and the specific §6.2.1.3 requirement for _each_ problem report could be more explicit.

**Strengths**:

- Safety evaluation explicitly required in §6.13
- Risk management process integration in maintenance plan
- Change decision requirement (change request or rationale for no action)

**Gaps**:

- §6.10 does not explicitly require that _each_ problem report undergo safety evaluation
- No direct cross-reference from §6.10 to §6.13's safety evaluation requirement

**Recommendation**: Strengthen §6.10 with explicit language: "The Software Maintenance Plan shall ensure that each Problem Report is evaluated to determine its effect on safety of released medical device software and whether a software change is needed to address the problem (see Section 6.13)."

### 6.2.3 Analyze Change Requests

**Standard Requirement**: In addition to the analysis required by Clause 9, the manufacturer shall analyze each change request for its effect on the organization, medical device software released for intended use, and systems with which it interfaces. [Class A, B, C]

**SOP Implementation** (§6.10, §6.13):
Section 6.13 requires that anomalies "shall be investigated, evaluated for safety, and will result in a change request or rationale for taking no action." However, neither §6.10 nor §6.13 explicitly requires analysis of change requests for effects on (1) the organization, (2) released software, and (3) interfacing systems.

**Compliance Rating**: ⚠️ **PARTIAL COMPLIANCE (60%)**

**Analysis**: The SOP establishes change request generation through the problem resolution process but lacks explicit requirements for the comprehensive impact analysis specified in IEC 62304:2006 §6.2.3. The standard requires three distinct dimensions of analysis: organizational impact, released software impact, and interface system impact. The SOP's safety evaluation requirement (§6.13) partially addresses software impact but does not mandate the breadth of analysis required. This gap could result in inadequate consideration of organizational resources, deployment complexity, or interface compatibility when evaluating maintenance changes.

**Gaps**:

- No explicit requirement to analyze organizational effects
- No explicit requirement to analyze effects on released software
- No explicit requirement to analyze effects on interfacing systems
- Missing "in addition to Clause 9 analysis" language indicating supplementary analysis beyond problem resolution

**Recommendation**: Add to §6.10 or §6.13: "Each change request arising from the maintenance process shall be analyzed for its effect on: (a) the organization (resources, competencies, infrastructure); (b) medical device software released for intended use (functionality, performance, safety); and (c) systems with which the software interfaces (compatibility, data exchange, dependencies)."

### 6.2.4 Change Request Approval

**Standard Requirement**: The manufacturer shall evaluate and approve change requests which modify released medical device software. [Class A, B, C]

**SOP Implementation** (§6.10, §6.13):
Section 6.13 indicates that anomalies "will result in a change request or rationale for taking no action" and that "any actions taken to resolve the anomaly shall be verified." However, neither §6.10 nor §6.13 explicitly establishes a formal change request evaluation and approval requirement for modifications to released software.

**Compliance Rating**: ⚠️ **PARTIAL COMPLIANCE (55%)**

**Analysis**: The SOP establishes change request generation and action verification but lacks explicit requirements for formal evaluation and approval of change requests _before_ implementation. IEC 62304:2006 §6.2.4 mandates a gate-keeping function where change requests modifying released software undergo evaluation and formal approval. The SOP's current language suggests that changes may proceed from problem investigation directly to action without a formal approval decision point, which could result in unauthorized modifications to released software. This is a significant gap in change control governance.

**Gaps**:

- No explicit change request approval requirement
- No specification of approval criteria or authority
- No formal gate between change request generation and implementation
- Absence of approval documentation requirements

**Recommendation**: Add to §6.10: "The Software Maintenance Plan shall establish procedures for evaluating and approving change requests that modify released medical device software. Approval criteria shall consider safety impact, regulatory requirements, resource availability, and interface effects. Approval authority and documentation requirements shall be defined."

### 6.2.5 Communicate to Users and Regulators

**Standard Requirement**: The manufacturer shall identify approved change requests affecting released medical device software. As required by local regulation, the manufacturer shall inform users and regulators about: (a) any problem in released medical device software and consequences of continued unchanged use; and (b) the nature of available changes and how to obtain and install them. [Class A, B, C]

**SOP Implementation** (§6.10, paragraph 3):
The SOP states: "As required by local regulation, the manufacturer may need to inform users and regulators regarding any problem with the released software and consequence of continued use along with the nature of any available software, how to obtain, and install these changes."

**Compliance Rating**: ✅ **SUBSTANTIAL COMPLIANCE (85%)**

**Analysis**: The SOP addresses both required communication elements: (1) problem notification and consequences of continued use, and (2) available changes and their installation. The scoping to "as required by local regulation" appropriately reflects the standard's conditional language. However, the SOP uses "may need to inform" language which could be interpreted as discretionary, whereas the standard's phrasing ("shall inform...as required by local regulation") indicates a mandatory obligation when regulatory requirements exist. Additionally, the SOP does not explicitly require _identifying_ approved change requests affecting released software, which is the first part of §6.2.5.

**Strengths**:

- Addresses both communication elements (problems and changes)
- Includes consequences of continued use
- Specifies obtaining and installing changes
- Appropriately scoped to regulatory requirements

**Gaps**:

- "May need to inform" language suggests discretion rather than obligation
- Does not explicitly require identifying approved change requests affecting released software
- No cross-reference to regulatory affairs or quality functions for communication execution

**Recommendation**: Revise language to: "The Software Maintenance Plan shall establish procedures to identify approved change requests affecting released medical device software. As required by local regulation, the manufacturer shall inform users and regulators regarding: (a) any problem in released medical device software and the consequences of continued unchanged use; and (b) the nature of any available changes to released medical device software and how to obtain and install the changes."

### 6.3.1 Use Established Process to Implement Modification

**Standard Requirement**: The manufacturer shall identify and perform any Clause 5 activities that need to be repeated as a result of the modification. [Class A, B, C]

**SOP Implementation** (§6.10, §6.13):
No explicit requirement found in §6.10 or §6.13 mandating identification and repetition of Clause 5 activities for maintenance modifications. Section 6.13 requires that "any actions taken to resolve the anomaly shall be verified" but does not reference Clause 5 development process activities.

**Compliance Rating**: ❌ **SIGNIFICANT GAP (40%)**

**Analysis**: This represents a significant gap in the SOP. IEC 62304:2006 §6.3.1 requires that maintenance modifications follow the same rigorous development processes as initial development, with appropriate selection and execution of Clause 5 activities (planning, requirements, architecture, detailed design, unit implementation, integration testing, system testing, release). The SOP's generic "verification" requirement (§6.13) does not substitute for systematic determination of which Clause 5 activities must be repeated based on modification scope and impact. Without this requirement, maintenance changes could bypass critical development and verification steps, increasing risk of introducing defects or compromising safety.

**Gaps**:

- No requirement to identify applicable Clause 5 activities for modifications
- No requirement to perform identified Clause 5 activities
- No guidance on determining which activities need repetition based on modification scope
- Missing linkage between maintenance modifications and development process requirements (§6.2-§6.9)

**Recommendation**: Add to §6.10: "The Software Maintenance Plan shall establish procedures to identify and perform Clause 5 activities (Sections 6.2-6.9 of this procedure) that need to be repeated as a result of modifications to released software. The extent of repeated development activities shall be determined based on modification scope, affected software items, and safety classification."

### 6.3.2 Re-Release Modified Software System

**Standard Requirement**: The manufacturer shall release modifications according to 5.8 (Software Release). Modifications can be released as a full re-release or as a modification kit with changed software items and installation tools. [Class A, B, C]

**SOP Implementation** (§6.9):
Section 6.9 (Software Release) provides comprehensive release requirements including release documentation, known anomaly evaluation, and release procedure/environment documentation. However, §6.10 (Software Maintenance) does not explicitly reference §6.9 or state that modifications shall be released according to established release procedures.

**Compliance Rating**: ⚠️ **PARTIAL COMPLIANCE (75%)**

**Analysis**: The SOP provides robust software release procedures in §6.9 but does not explicitly connect maintenance modifications to those release procedures within §6.10. This creates ambiguity about whether maintenance releases must follow the same release controls as initial releases. While organizational practice may assume this connection, the absence of explicit linkage in the maintenance section risks inconsistent release practices for modifications. IEC 62304:2006 §6.3.2's explicit mandate to release modifications "according to 5.8" ensures that modifications undergo the same release rigor as original software, including documentation, verification, and delivery controls.

**Strengths**:

- §6.9 provides comprehensive release requirements
- Release documentation and anomaly evaluation required
- Release procedures and environment documentation mandated

**Gaps**:

- §6.10 does not reference §6.9 or require modifications to follow release procedures
- No explicit statement that maintenance modifications undergo formal release process
- Missing guidance on modification kit releases vs. full re-releases

**Recommendation**: Add to §6.10: "Modifications to released software shall be released in accordance with Section 6.9 (Software Release). Modifications may be released as a full software system re-release or as a modification kit comprising changed software items and necessary installation tools."
