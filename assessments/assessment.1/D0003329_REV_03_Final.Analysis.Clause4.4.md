>

### 4.4.3 Gap Analysis

#### IEC 62304 Requirement

**Standard Text** (IEC 62304:2006+A1:2015, §4.4.3):

> "Based on the software safety class of the LEGACY SOFTWARE (see 4.3), the MANUFACTURER shall perform a gap analysis of available DELIVERABLES against those required according to 5.2, 5.3, 5.7, and Clause 7.
>
> a) The MANUFACTURER shall assess the continuing validity of available DELIVERABLES.
>
> b) Where gaps are identified, the MANUFACTURER shall EVALUATE the potential reduction in RISK resulting from the generation of the missing DELIVERABLES and associated ACTIVITIES.
>
> c) Based on this evaluation, the MANUFACTURER shall determine the DELIVERABLES to be created and associated ACTIVITIES to be performed. The minimum DELIVERABLE shall be SOFTWARE SYSTEM test records (see 5.7.5).
>
> NOTE: Such gap analysis should assure that RISK CONTROL measures, implemented in LEGACY SOFTWARE, are included in the software requirements."

**Interpretation**: Gap analysis is a four-step process:

1. Identify required deliverables based on software safety class (A, B, or C)
2. Assess which required deliverables exist and whether they remain valid
3. For missing deliverables, evaluate whether creating them would reduce risk
4. Decide which deliverables to create (minimum: system test records) based on risk reduction potential

The note emphasizes that any risk controls embedded in legacy software must be captured as requirements.

#### D0003329 Rev 03 Coverage

**Analysis**: The SOP acknowledges gap analysis at a superficial level but provides no procedural guidance. The single bullet point mentions the need for gap analysis but omits critical details about process, decision criteria, and mandatory activities.

### 4.4.5 Rationale for Use of Legacy Software

#### IEC 62304 Requirement

**Standard Text** (IEC 62304:2006+A1:2015, §4.4.5):

> "The MANUFACTURER shall document the VERSION of the LEGACY SOFTWARE together with a rationale for the continued use of the LEGACY SOFTWARE based on the outputs of 4.4.
>
> NOTE: Fulfilling 4.4 enables further use of LEGACY SOFTWARE in accordance with IEC 62304."

**Interpretation**: This requirement mandates comprehensive documentation that synthesizes all 4.4 activities into a justified decision to use legacy software. The rationale must be evidence-based, drawing from:

- Post-production surveillance (4.4.2a)
- Risk assessment (4.4.2b)
- Gap analysis results (4.4.3)
- Gap closure plan and execution (4.4.4)

The version identification ensures traceability to specific software configuration.

#### D0003329 Rev 03 Coverage

**SOP Text** (Section 6.1, Lines 223):

> "- Document the version of the legacy software with a rationale for the continued use."

**Analysis**: The SOP acknowledges the requirement but provides no guidance on rationale content, structure, approval, or integration into design history file.

- Mentions both version documentation and rationale
