## IEC 62304 Clause 7 Requirements Analysis

### Core Requirements Structure

IEC 62304 Clause 7 establishes a software-specific risk management process embedded within the device-level ISO 14971 framework:

1. **Section 7.1 - Analysis of Software Contributing to Hazardous Situations**
   - 7.1.1: Identify software items that could contribute to hazardous situations (Class B, C)
   - 7.1.2: Identify potential causes (incorrect specs, defects, SOUP failures, hardware failures, foreseeable misuse) (Class B, C)
   - 7.1.3: Evaluate published SOUP anomaly lists against known hazards (Class B, C)
   - 7.1.4: Document potential causes in Risk Management File (Class B, C)

2. **Section 7.2 - Risk Control Measures**
   - 7.2.1: Define and document risk control measures per ISO 14971 (Class B, C)
   - 7.2.2: For software-implemented controls: include in requirements, assign safety class, develop per Clause 5 (Class B, C)

3. **Section 7.3 - Verification of Risk Control Measures**
   - 7.3.1: Verify implementation, review for new hazardous situations (Class B, C)
   - 7.3.3: Document traceability (hazardous situation → software item → cause → control → verification) (Class B, C)

4. **Section 7.4 - Risk Management of Software Changes**
   - 7.4.1: Analyze changes for new potential causes (Class A, B, C)
   - 7.4.2: Analyze impact on existing risk controls (Class B, C)
   - 7.4.3: Perform 7.1-7.3 activities as needed (Class B, C)

**Standard Expectation**: Systematic identification, causation analysis, control definition, verification, and change impact assessment with full traceability.

## Section 6.11 Coverage Assessment

### Strengths

1. **ISO 14971 Integration**: Correctly establishes that software risk analysis follows the project Risk Management Plan (references D0003103).
2. **Hazard Identification**: Explicitly requires identification of "software items that may contribute to a hazardous situation involving software, including SOUP."
3. **Risk Control in Requirements**: States that risk control measures implemented in software "shall be included in the Software Requirements."
4. **Safety Classification**: Requires assignment of software safety class based on risk severity.
5. **Traceability (Class B/C)**: Mandates traceability documentation from hazardous situation → software item → cause → risk control → verification.
6. **Cybersecurity Integration**: Adds data/security risk management process with reference to D0029257.

### Critical Gaps

#### **1. Missing Causation Analysis (7.1.2)**

- **Gap**: Section 6.11 does not instruct teams to systematically analyze **potential causes** (incorrect specs, software defects, SOUP failures, hardware failures, foreseeable misuse).
- **Impact**: Teams may identify hazardous software items but fail to explore root causes, leading to incomplete risk control strategies.
- **Severity**: **CRITICAL** — Core requirement for Classes B and C.

#### **2. No SOUP Anomaly List Evaluation (7.1.3)**

- **Gap**: Missing explicit requirement to evaluate published SOUP anomaly lists against identified hazards.
- **Impact**: Known vulnerabilities in third-party components (libraries, frameworks) may go undetected.
- **Severity**: **HIGH** — Particularly critical for connected medical devices with extensive SOUP dependencies.

#### **3. Absent Lifecycle Integration Workflow**

- **Gap**: Section 6.11 does not specify **when** in the SDLC these activities occur or how they integrate with:
  - Requirements analysis (5.2)
  - Architecture design (5.3)
  - Detailed design (5.4)
  - System testing (5.7)
  - Software changes (6.1, 7.4)
- **Impact**: Risk management becomes a disconnected checklist rather than an embedded lifecycle process.
- **Severity**: **CRITICAL** — Without phase-specific touchpoints, risk management occurs ad-hoc or too late.

#### **4. Incomplete Verification Process (7.3.1)**

- **Gap**: While traceability is mentioned, the SOP does not detail **how** risk control measures are verified (test methods, acceptance criteria, regression requirements).
- **Impact**: Verification becomes subjective; insufficient evidence for regulatory review.
- **Severity**: **HIGH** — Essential for demonstrating effective risk control.

#### **5. Inadequate Change Management Integration (7.4)**

- **Gap**: Section 6.11 identifies new hazards from changes but does not explicitly require:
  - Analysis of changes for new potential causes (7.4.1)
  - Impact assessment on existing risk controls (7.4.2)
  - Re-execution of 7.1-7.3 activities (7.4.3)
- **Impact**: Software updates may inadvertently disable risk controls or introduce new hazards without detection.
- **Severity**: **HIGH** — Post-market changes are high-risk modification points.

#### **6. No Guidance on New Hazard Review (7.3.1 Note)**

- **Gap**: The standard requires review to determine if risk control measures could introduce **new hazardous situations**. Section 6.11 mentions identifying "any new sequence of events" but does not embed this in a structured verification workflow.
- **Impact**: Controls may mitigate one hazard while creating another (e.g., timeout safety feature causing data loss).
- **Severity**: **MODERATE** — Regulatory inspectors frequently audit this requirement.

## High-Priority Recommendations

### Recommendation 1: Add Causation Analysis Procedure (CRITICAL)

**Action**: Expand Section 6.11 to require systematic analysis of potential causes per IEC 62304:7.1.2:

- Incorrect or incomplete specifications
- Software defects in functionality
- SOUP failure or unexpected results
- Hardware failures leading to unpredictable software behavior
- Reasonably foreseeable misuse

**Implementation**: Add subsection "6.11.2 Causation Analysis" with checklist/template reference.

### Recommendation 2: Mandate SOUP Anomaly List Review (HIGH)

**Action**: Insert requirement:

> "For SOUP items identified as contributing to hazardous situations, the development team shall review the supplier's published anomaly list for the specific version in use and document any known anomalies that could lead to hazards."

**Implementation**: Reference SOUP registry template (if available) or create appendix with review log format.

### Recommendation 3: Define SDLC Integration Points (CRITICAL)

**Action**: Create process flow diagram or table mapping risk management activities to SDLC phases:

- **Requirements (5.2)**: Risk control measures → requirements
- **Architecture (5.3)**: Safety class assignment to components
- **Detailed Design (5.4)**: Design-level risk control implementation
- **Unit/Integration Testing (5.5-5.6)**: Risk control verification (partial)
- **System Testing (5.7)**: Full risk control verification (7.3.1)
- **Maintenance (6.1)**: Change impact analysis (7.4)

**Implementation**: Add "6.11.1 Risk Management Lifecycle Integration" subsection with phase-activity matrix.

### Recommendation 4: Specify Verification Methods (HIGH)

**Action**: Detail verification evidence requirements:

- Test cases demonstrating risk control effectiveness
- Acceptance criteria for safety-critical functions
- Regression test requirements for risk controls
- Documentation in test reports linking to Risk Management File

**Implementation**: Reference test plan template or add Appendix with verification checklist.

### Recommendation 5: Strengthen Change Management Integration (HIGH)

**Action**: Add explicit subsection "6.11.5 Risk Management for Software Changes":

- Require analysis per 7.4.1 (new causes)
- Require analysis per 7.4.2 (impact on existing controls)
- Require re-execution of 7.1-7.3 activities where applicable
- Link to Change Control process (Section 8)

**Implementation**: Cross-reference with D0003103 change management procedures.

### Recommendation 6: Add New Hazard Review Step

**Action**: Require that during risk control verification (7.3.1), teams explicitly assess whether the control measure itself could introduce new hazardous situations.

**Implementation**: Add to verification checklist: "Does this risk control measure create any new failure modes or hazardous situations?"
