## IEC 62304 Clause 9 Requirements Summary

IEC 62304 Clause 9 establishes eight mandatory requirements applicable to all software safety classes (A, B, C):

1. **9.1 Prepare Problem Reports**: Document problems with criticality statements and supporting information
2. **9.2 Investigate Problems**: Analyze root causes, evaluate safety relevance, document outcomes, generate change requests or no-action rationale
3. **9.3 Advise Relevant Parties**: Communicate problem existence appropriately
4. **9.4 Use Change Control Process**: Implement approved change requests through configuration management
5. **9.5 Maintain Records**: Preserve problem reports, resolutions, verifications; update risk management files
6. **9.6 Analyse Problems for Trends**: Detect patterns in problem occurrence
7. **9.7 Verify Problem Resolution**: Confirm problem closure, trend reversal, implementation completeness, absence of new problems
8. **9.8 Test Documentation Contents**: Document test results, anomalies, versions, configurations, tools, dates, and tester identification

## Coverage Assessment

### Strengths

**Lifecycle Integration (9.1, 9.4, 9.5)**
Section 6.13 effectively establishes problem resolution across the software lifecycle by referencing the Software Development Plan for developmental anomalies and the Software Maintenance Plan for production issues. This two-phase approach aligns with IEC 62304's expectation that problem resolution operates continuously. The requirement to "log and track" complaints requiring software action (paragraph 2) satisfies basic record maintenance per requirement 9.5.

**Investigation and Safety Evaluation (9.2)**
The SOP explicitly requires that "all anomalies identified shall be investigated, evaluated for safety" (paragraph 3). This directly addresses IEC 62304 §9.2's core mandate to investigate problems and evaluate safety relevance using the risk management process. The requirement for either a "change request or rationale for taking no action" aligns with §9.2(d)'s documentation expectations.

**Quality System Integration (9.3, 9.4)**
The reference to D0003325 (Global Procedure, Nonconformance) for "software related complaint escalation" (paragraph 2) provides implicit coverage of §9.3's requirement to advise relevant parties. Integration with change control processes is implied through references to Software Development and Maintenance Plans, satisfying §9.4's structural requirement.

**Verification Requirement (9.7 - Partial)**
The statement that "actions taken to resolve the anomaly shall be verified and ensure the change did not result in additional anomalies" (paragraph 3) addresses two of §9.7's four verification criteria: problem closure (9.7a) and absence of introduced problems (9.7d).

### Gaps

**Trend Analysis (9.6) - Critical Gap**
Section 6.13 contains **no explicit requirement** for analyzing problems to detect trends. This represents a critical omission since IEC 62304 §9.6 mandates trend analysis for all software safety classes. Trend analysis enables proactive identification of systemic issues before they escalate to safety-critical failures. The standard's guidance (Annex B.9) emphasizes that recognizing patterns is a core objective of the problem resolution process.

**Comprehensive Verification (9.7) - Moderate Gap**
While verification is mentioned, §6.13 addresses only two of four mandatory verification elements:

- **Missing**: Verification that adverse trends have been reversed (§9.7b)
- **Missing**: Explicit verification that change requests have been implemented in appropriate medical device software and activities (§9.7c)

The current language focuses on individual anomaly resolution without confirming systematic implementation across affected systems or reversing negative patterns.

**Test Documentation Requirements (9.8) - Significant Gap**
Section 6.13 makes **no reference** to IEC 62304 §9.8's seven mandatory test documentation elements:

- Test results
- Anomalies found
- Software version tested
- Hardware/software test configurations
- Test tools used
- Test date
- Tester identification

This omission is particularly significant because §9.8 applies to testing, retesting, and regression testing following changes—activities central to problem resolution verification.

**Problem Report Content (9.1) - Minor Gap**
While implied through template references (D0004198, Global Template, Software Anomaly Report), §6.13 does not explicitly require problem reports to include "criticality statements" or "other information that may aid in resolution" such as affected devices or accessories, as specified in §9.1.

## Risk Management and CAPA Integration

Section 6.13's mandate to "evaluate for safety" (paragraph 3) implicitly invokes the risk management process required by IEC 62304 §9.2(b). However, the section does not explicitly reference updating the risk management file per §9.5's requirement. The integration with D0003325 (Nonconformance) suggests connection to corrective and preventive action (CAPA) systems, but this linkage is indirect rather than procedurally explicit.

The SOP would benefit from explicit statements requiring risk management file updates when safety-relevant problems are identified and documented CAPA linkage to ensure closed-loop feedback from problem detection through systemic correction.

## Recommendations

### High Priority

1. **Add Trend Analysis Requirement**: Insert explicit language requiring periodic analysis of problem reports to detect patterns, including frequency of analysis and responsibility assignment. Consider referencing post-market surveillance integration per D0004078 and D0004104.

2. **Expand Verification Requirements**: Enumerate all four §9.7 verification criteria, particularly trend reversal confirmation and systematic implementation verification across affected software items.

3. **Incorporate Test Documentation Standards**: Either embed §9.8's seven documentation requirements directly in §6.13 or reference D0017982/D0017983 (Software Verification Protocol/Report templates) as mandatory frameworks for post-resolution testing.

### Moderate Priority

4. **Clarify Problem Report Content**: Explicitly require criticality statements and resolution-aiding information (affected products, configurations, etc.) in problem reports, referencing D0004198 template as the implementation mechanism.

5. **Formalize Risk Management Linkage**: Add explicit requirement to update the risk management file when safety-relevant problems are identified, per §9.5.

6. **Document CAPA Integration**: Expand D0003325 reference to explicitly describe how software problem resolution feeds into and receives feedback from the CAPA system.

### Low Priority

7. **Cross-Reference Enhancement**: Add forward references from §6.13 to related sections (§6.12 Configuration Management, §6.10 Maintenance Planning) to improve procedural clarity.
