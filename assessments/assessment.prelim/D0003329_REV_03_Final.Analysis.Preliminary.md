
# Compliance Review: D0003329 Rev 03 Final - Software Development Work Instruction

## Intro

This preliminary assessment provides a detailed compliance review of the Global Work Instruction for Software Development (D0003329 Rev 03) against the IEC 62304 standard (BS EN 62304), the parent Design Control Process (D0003098), and internal consistency checks. The goal is to identify gaps, ambiguities, and areas needing enhancement to ensure full regulatory compliance and effective integration with overall design controls.

The assessment relied on the LLM's understanding of IEC 62304 requirements, FDA design control expectations, and best practices in medical device software development, not the actual 62304 standard text.

Specific sections of D0003329 were analyzed in detail, with findings categorized into compliant elements, partial compliance/clarifications needed, and non-compliant/missing elements.

## **Compliance Review: D0003329 Rev 03 - Global Work Instruction, Software Development**

Based on comprehensive review of D0003329 against IEC 62304 (BS EN 62304), D0003098, and D0003329 self-consistency, here are the compliance findings:

---

### **Executive Summary**

D0003329 Rev 03 provides a **generally compliant** framework for IEC 62304-based software development with appropriate mappings to standard clauses. However, several **critical gaps** exist in detailed requirements, verification criteria, and integration with parent design control process D0003098. Key concerns include insufficient SOUP management detail, incomplete verification requirements specification, and ambiguous integration between software-specific and system-level activities.

**Overall Assessment**: ⚠️ **Partially Compliant** - Requires targeted enhancements to achieve full regulatory compliance.

---

### **Compliance Findings**

#### **✅ COMPLIANT ELEMENTS**

**1. Structure and Organization**
- Clear mapping to BS EN 62304 sections (4.4, 5.1-5.8, 6.1, 7, 8, 9)
- Appropriate software safety classification framework (Appendix 1)
- References to appropriate templates (D0004168-D0004199, D0017982-D0017986)

**2. Software Safety Classification (Section 6.1 & Appendix 1-2)**
- Defines Class A/B/C based on risk (IEC 62304 §4.3)
- Classification process involving R&D and Clinical Medical Affairs
- Documentation requirement in risk management file

**3. Legacy Software (Section 6.1)**
- Addresses IEC 62304 §4.4 with two compliant paths
- Gap analysis requirement for missing deliverables
- Risk management integration for continued use

**4. Software Development Planning (Section 6.2)**
- Addresses IEC 62304 §5.1 comprehensively
- Defines required plan elements including deliverables, verification, risk management
- Class-specific requirements for B/C devices (unit testing, integration, common defects)

**5. Risk Management (Section 6.11)**
- Addresses IEC 62304 §7 with risk analysis requirements
- Hazardous situation identification including SOUP
- Risk control measure traceability (Class B/C)
- Reference to cybersecurity (D0029257)

**6. Configuration Management (Section 6.12)**
- Addresses IEC 62304 §8 with CM plan requirement
- Covers tools, change control, SOUP handling, configuration identification
- Repository requirements with audit trail

---

#### **⚠️ PARTIAL COMPLIANCE / NEEDS CLARIFICATION**

**1. SOUP Management (Multiple Sections)**

*Issue*: SOUP requirements are scattered and lack comprehensive guidance.

*Specific Concerns*:
- Section 6.4 (Architecture) mentions SOUP functional/performance requirements and support requirements but doesn't specify **how** to establish these requirements
- No explicit requirement for SOUP Bill of Materials or SOUP list
- Missing guidance on SOUP version control and obsolescence monitoring process
- Software Maintenance Plan (Section 6.10) mentions "upgrades, bug fixes, patches, and obsolescence of SOUP" but doesn't define **evaluation criteria**

*IEC 62304 Gap*: §5.1.1(d), §5.3.3, §5.3.4, §8.1.2 require detailed SOUP identification and control

*Recommendation*: Add explicit SOUP management section or expand Section 6.4 to include:
- SOUP identification and documentation requirements (name, version, manufacturer)
- SOUP risk assessment process
- SOUP functional/performance requirements specification method
- SOUP anomaly list maintenance
- SOUP obsolescence monitoring frequency and triggers

---

**2. Software Requirements (Section 6.3)**

*Issue*: Requirements management process lacks sufficient detail on verification criteria.

*Specific Concerns*:
- States requirements "permit establishment of test criteria" but doesn't define **what constitutes adequate test criteria**
- No explicit requirement for functional vs. non-functional requirements categorization
- Missing guidance on requirements attributes (priority, traceability ID, verification method)
- Doesn't specify **when** requirements review should occur in the lifecycle

*IEC 62304 Gap*: §5.2.2, §5.2.6 require re-evaluation and content verification

*Recommendation*: Enhance Section 6.3 to include:
- Requirements review timing (e.g., before architecture, at design reviews)
- Minimum requirements attributes (ID, description, rationale, verification method, safety class)
- Functional vs. non-functional requirements distinction
- Requirements change control process reference

---

**3. Integration with D0003098 Design Control Process**

*Issue*: Unclear alignment between D0003329 software activities and D0003098 7-phase process.

*Specific Concerns*:
- D0003098 Phase 2 states "software requirements shall be developed from system requirements" (line ~417) but D0003329 doesn't specify **which phase** software planning occurs
- D0003098 Phase 3 mentions "Software Development Outputs" developed per D0003329 but doesn't map specific D0003329 sections to Phase 3 activities
- D0003098 Phase 4 addresses Design Verification but doesn't clarify relationship to D0003329 Section 6.7 (Integration Testing) and 6.8 (System Testing)
- Risk management integration: D0003098 references D0003349 while D0003329 references "overall project Risk Management Plan" without cross-reference

*Compliance Gap*: FDA Design Control requires integrated system-level and software-level processes

*Recommendation*: Add integration matrix showing:
- D0003098 Phase → D0003329 Section mapping
- Design review coordination (Phase Reviews vs. Software Reviews)
- DHF content from software development (which D0003329 outputs go into DHF)
- Traceability between system requirements (D0003335) and software requirements (D0004169)

---

**4. Software Detailed Design (Section 6.5)**

*Issue*: Requirements for Class C devices are incomplete.

*Specific Concerns*:
- States "software detailed design shall include a description of interfaces" but only for Class C
- Review requirement only applies to Class C, creating potential gap for Class B
- No guidance on **level of detail** required for software unit design
- Missing cross-reference to common software defects identified in SDP (Section 6.2)

*IEC 62304 Gap*: §5.4.1 requires detailed design "to the software unit level" for all Class B/C

*Recommendation*: Clarify Class B detailed design expectations and add:
- Minimum content requirements (algorithms, data structures, interfaces)
- Design patterns or standards to be followed
- Relationship between detailed design and unit implementation (Section 6.6)

---

**5. Software Unit Implementation (Section 6.6)**

*Issue*: Unit testing requirements are vague.

*Specific Concerns*:
- "Unit-level testing shall be completed with the level of detail based on the risk" - no objective criteria for "level of detail"
- Class C acceptance criteria listed as "should be considered" (not mandatory language)
- No requirement for **unit test documentation** or pass/fail criteria
- Missing guidance on code review methods (peer review, walkthrough, inspection)

*IEC 62304 Gap*: §5.5.5 requires verification that units are "free from unacceptable anomalies"

*Recommendation*: Strengthen Section 6.6 with:
- Mandatory unit test documentation requirement
- Code review acceptance criteria (e.g., critical defects = 0, coding standard compliance)
- Unit test coverage expectations (Class B/C)
- Reference to coding standards mentioned in SDP (Section 6.2)

---

**6. Traceability (Section 6.8)**

*Issue*: Traceability requirements are mentioned only in System Testing section, not as standalone requirement.

*Specific Concerns*:
- Traceability mentioned in Section 6.8 but not defined as overall requirement
- Doesn't specify **bidirectional** traceability requirement
- No guidance on traceability matrix maintenance throughout lifecycle
- Missing traceability to risk controls (mentioned in Section 6.11 but not integrated with Section 6.8)

*IEC 62304 Gap*: §5.7.4, §5.7.5 require traceability including anomaly lists

*Recommendation*: Create standalone Traceability section or expand Section 6.8 to mandate:
- Bidirectional traceability: System Requirements ↔ Software Requirements ↔ Architecture ↔ Detailed Design ↔ Code ↔ Unit Tests ↔ Integration Tests ↔ System Tests
- Risk Control Measures → Requirements → Design → Verification
- Traceability matrix template reference (D0017985) with required fields

---

#### **❌ NON-COMPLIANT OR MISSING ELEMENTS**

**1. Software Verification Planning - Missing Acceptance Criteria (Section 6.8)**

*Critical Gap*: Section 6.8 states "Software Verification Protocols shall be created" but doesn't define **acceptance criteria** for test results.

*IEC 62304 Requirements*: §5.7.1 requires establishing criteria for software system test pass/fail

*Impact*: Without defined acceptance criteria, verification may be subjective and fail regulatory scrutiny

*Recommendation*: Add to Section 6.8:
```
Software Verification acceptance criteria shall be established prior to test execution and shall include:
- Pass/fail criteria for each test case
- Acceptable anomaly severity thresholds (e.g., no Critical defects, <5 Major defects)
- Performance acceptance criteria (response time, throughput, resource utilization)
- Regression test pass rate (e.g., 100% of previously passing tests must pass)
```

*Standard Citation*: IEC 62304 §5.7.1(a), §5.7.3

---

**2. Problem Resolution Process - Insufficient Detail (Section 6.13)**

*Critical Gap*: Section 6.13 is extremely brief (5 sentences) and lacks procedural detail required by IEC 62304 §9.

*Missing Elements*:
- No process for **analyzing** problems (root cause analysis)
- No **prioritization criteria** for problem resolution
- No **documentation requirements** beyond "logged and tracked"
- No **change control integration** for fixes
- No **verification of fixes** process (mentioned in one sentence but not detailed)
- No **trend analysis** requirement

*IEC 62304 Requirements*:
- §9.2: Use software configuration management process for implementing changes
- §9.4: Verify changes (regression testing)
- §9.5: Test documentation update
- §9.7: Document problems and track to resolution
- §9.8: Analyze problems for trends

*Impact*: High - Problem resolution is critical for both development and post-market phases

*Recommendation*: Expand Section 6.13 to include:
```
Problem Analysis (IEC 62304 §9.2-9.3):
- Root cause analysis requirement
- Safety impact assessment
- Severity classification (Critical, Major, Minor)

Problem Resolution:
- Prioritization criteria based on severity and safety impact
- Change request creation and approval
- Implementation via configuration management process (Section 6.12)

Verification of Changes (IEC 62304 §9.4-9.5):
- Regression test requirements
- Verification that fix resolves problem without introducing new issues
- Test documentation updates

Trend Analysis (IEC 62304 §9.8):
- Periodic review of problem patterns
- Corrective and preventive action (CAPA) initiation criteria
```

*Standard Citation*: IEC 62304 §9.1-9.8 (entire section)

---

**3. Software Maintenance Plan - Missing Post-Market Requirements**

*Critical Gap*: Section 6.10 addresses Software Maintenance Plan but is **missing critical FDA post-market requirements**.

*Missing Elements*:
- No **post-market surveillance data integration** (adverse events, complaints)
- No **software update validation** requirements before release
- No **user notification process** for updates (only mentions "may need to inform")
- No **version management** for fielded software
- No **rollback procedure** if update causes issues

*D0003098 Integration Gap*: D0003098 references D0003293 (Post-Market Surveillance) but D0003329 doesn't integrate PMS data into maintenance

*Regulatory Impact*: FDA 21 CFR 820.30(j) requires design changes to be verified/validated

*Recommendation*: Enhance Section 6.10 to include:
```
Post-Market Surveillance Integration:
- Complaint evaluation per D0003325
- PMS data review per D0003293
- Adverse event trending and analysis

Software Update Process:
- Change impact assessment per D0003336
- Regression verification requirements (full vs. targeted based on risk)
- Validation requirements for patches/updates
- Version Description Document (D0004199) for each release

User Communication:
- Mandatory notification for safety-related updates
- Update installation instructions
- Known issue documentation
- Fielded version tracking

Rollback Procedures:
- Criteria for rollback decision
- Rollback testing requirements
- User communication for rollback scenarios
```

*Standard Citation*: IEC 62304 §6.2-6.3, FDA Guidance on Software Validation

---

**4. Design Reviews - Absent Section**

*Critical Gap*: D0003329 **never mentions design reviews** despite multiple references in D0003098 to Phase Design Reviews.

*IEC 62304 Requirement*: §5.1.1(e) requires software development plan to include "reviews for VERIFICATION of the software development plan"

*D0003098 Context*:
- Phases 1-4 all require Phase Design Reviews per section 6.9
- Software is part of design outputs (Phase 3)
- D0003098 states "A formal design review shall be conducted to confirm completion of software deliverables" (line ~333)

*Impact*: Software development may proceed without required milestone reviews

*Recommendation*: Add new Section 6.14 "Software Design Reviews":
```
6.14 Software Design Reviews (IEC 62304 §5.1.6)

Software design reviews shall be conducted at the following minimum milestones:
- Software Development Plan approval
- Software Requirements completion (before Architecture)
- Software Architecture approval (Class B/C, before Detailed Design)
- Software Detailed Design approval (Class B/C, before Implementation)
- Software System Test completion (before Release)

Design reviews may be combined with D0003098 Phase Design Reviews as appropriate.

Review participants shall include:
- R&D development team
- Design Assurance / Quality
- Independent Reviewer (as defined in Responsibilities table)
- Subject Matter Experts as needed

Review criteria shall confirm:
- Completeness of phase deliverables per Software Development Plan
- Compliance with IEC 62304 requirements for software safety class
- Traceability adequacy
- Risk control measure implementation
- Anomaly resolution status

Review results shall be documented with action items tracked to closure.
```

*Standard Citation*: IEC 62304 §5.1.6, §5.1.9

---

**5. Network Security / Cybersecurity Requirements - Inadequate Detail**

*Concern*: Section 6.11 references D0029257 (Cybersecurity Work Instruction) but provides **minimal integration guidance**.

*Missing Elements*:
- No requirement for **threat modeling** in architecture phase
- No **security testing** requirements in verification (Section 6.8)
- No **vulnerability assessment** in maintenance (Section 6.10)
- No **security risk controls** traceability

*FDA Guidance*: FDA Premarket Cybersecurity and Postmarket Cybersecurity Management require comprehensive security controls

*Recommendation*: Expand Section 6.11 cybersecurity paragraph to mandate:
- Threat modeling as part of Software Architecture (Section 6.4)
- Security requirements in Software Requirements (Section 6.3)
- Security test cases in Software Verification Protocol (Section 6.8)
- Vulnerability monitoring in Software Maintenance Plan (Section 6.10)
- Reference D0029257 for detailed cybersecurity requirements

*Standard Citation*: IEC 62304 §7 (risk management), FDA Cybersecurity Guidance

### **Recommendations (Prioritized)**

#### **Critical Priority (Immediate Action Required)**

1. **Add Section 6.14 - Software Design Reviews** (IEC 62304 §5.1.6)
   - Define mandatory review milestones
   - Specify review participants and criteria
   - Integrate with D0003098 Phase Design Reviews

2. **Expand Section 6.8 - Define Verification Acceptance Criteria** (IEC 62304 §5.7.1)
   - Pass/fail criteria for test cases
   - Anomaly severity thresholds
   - Performance acceptance limits

3. **Rewrite Section 6.13 - Problem Resolution Process** (IEC 62304 §9)
   - Problem analysis, prioritization, resolution, verification
   - Trend analysis requirement
   - Integration with configuration management

#### **High Priority (Required for Full Compliance)**

4. **Add Section 6.15 - SOUP Management** or expand Section 6.4
   - SOUP identification, version control, risk assessment
   - SOUP Bill of Materials requirement
   - Obsolescence monitoring process

5. **Create Traceability Section** or expand Section 6.8
   - Bidirectional traceability requirements
   - Integration with risk control measures
   - Traceability matrix maintenance throughout lifecycle

6. **Enhance Section 6.10 - Software Maintenance**
   - Post-market surveillance integration
   - Software update validation requirements
   - User notification and rollback procedures

7. **Strengthen Section 6.6 - Unit Implementation**
   - Mandatory unit test documentation
   - Code review acceptance criteria
   - Test coverage expectations

#### **Medium Priority (Enhancements for Clarity)**

8. **Add D0003098 Integration Matrix**
   - Map 7-phase process to D0003329 sections
   - Clarify DHF content from software development
   - Coordinate design reviews

9. **Expand Section 6.3 - Software Requirements**
   - Requirements review timing
   - Requirements attributes specification
   - Functional vs. non-functional distinction

10. **Clarify Section 6.5 - Detailed Design (Class B)**
    - Level of detail expectations
    - Minimum content requirements

11. **Expand Section 6.11 - Cybersecurity Integration**
    - Security requirements, testing, maintenance activities
    - Clearer integration with D0029257

3. Update Section 6.8 to add explicit acceptance criteria table/checklist

**Training Implications:**
- Software development teams will need training on enhanced requirements (design reviews, acceptance criteria, problem resolution)
- Independent Reviewers need guidance on their expanded role in design reviews

**Template Updates:**
- D0004168 (SDP template) should be updated to reflect enhanced planning requirements
- D0017982 (Verification Protocol) should include acceptance criteria section
- Consider creating new D-series template for SOUP Management

**Regulatory Considerations:**
- Rev 04 changes may require notification to regulatory bodies if products are already approved
- FDA 510(k) submissions should reference updated work instruction
- EU Technical Documentation should be updated to reflect new procedures

This assessment identifies **3 Critical gaps**, **4 High priority gaps**, and **4 Medium priority gaps** requiring resolution for full IEC 62304 compliance and integration with D0003098 design control framework.
